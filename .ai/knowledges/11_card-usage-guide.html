drwn — Card Usage Guide

Scenarios for modifying, creating, and switching between Darwinian Harness Cards. Each scenario lists exact CLI commands and the disk-state transitions they cause.

The state model

drwn moves bits between five distinct layers. Knowing which layer a command touches is the fastest way to understand what's happening on disk.

Machine library
~/.agents/drwn/
machine.json
skills/
extensions/
Card source (editable)
~/.agents/drwn/sources/
<scope>/<name>/
card.json + skills/ + ...
Card store (immutable)
~/.agents/drwn/cards/

Project
<project>/.agents/drwn/

LayerMutabilityWhat it holds
Machine libraryYou editAuthoring scope, machine-wide skill / MCP defaults, extension state.
Card sourceYou editThe working copy of a card you're authoring. Mutable until you publish a version.
Card storeAppend-onlyBare git repos per card + integrity-hashed extracted snapshots per published version. Read-only by convention.
ProjectYou edit (via CLI)config.json declares cards + overlay; card.lock is the resolved snapshot.
DownstreamGeneratedSymlinks and JSON written by drwn write. Never edit by hand — the next write replaces it.

How drwn write works

Every consumer-side scenario ends with drwn write. Knowing the steps makes the rest of this guide concrete.

1.
Read project config.json + card.lock.
2.
Resolve each card ref against the card store (or file: source).
3.
Compute the effective skill + MCP set, layering: machine defaults → cards → project overlay.
4.
Plan symlinks under .claude/skills/, .codex/skills/, and generated MCP configs.
5.
Compare plan against current downstream state. Display as a diff with --dry-run; apply otherwise.
6.
Replace stale drwn-managed symlinks; preserve user-owned files with a warning.

Always preview with --dry-run before a real write — especially when switching cards or removing one, since the diff makes the replace/remove actions visible.

Scenario A · Iterating on your own card during development

Goal: edit a skill body or add a new skill to a card you already own. No version bump, no consumer impact, fast feedback loop.

Prereqs

  • Card source already exists at ~/.agents/drwn/sources/@scope/name/.
  • A scratch project with drwn card apply file:<source-dir> already run, so you have a place to test materialization.

Commands

# 1. Edit the SKILL.md in staging (or directly in the source dir).
$ vim /tmp/staging/<skill>/SKILL.md

# 2. Re-add into the source (overwrite the bundled copy).
$ drwn card source add-skill @scope/name <skill> \
    --from /tmp/staging/<skill> --replace

# 3. Validate the source.
$ drwn card source doctor @scope/name --json

# 4. Re-materialize in the scratch project (file: refs re-read each write).
$ cd /tmp/scratch-project
$ drwn write --dry-run
$ drwn write

# 5. Smoke test in a fresh Claude Code session (the skill cache loads at startup).
$ claude

What's happening under the hood

StepLayer touchedEffect
add-skill --replaceSourcecp -r overwrites sources/@scope/name/skills/<skill>/. card.json.skills.include already lists the name; no manifest change.
source doctorSource (read-only)Walks the dir, checks SKILL.md exists, validates the manifest, surfaces orphaned/missing entries.
drwn writeProject → DownstreamThe applied file: ref uses range *, so the source is re-read each write. The symlink target (~/.agents/drwn/sources/...) is the same path; only the content under it changed.
Restart ClaudeRuntimeClaude Code loads skills at session start. A running session won't see the edit.
Why no version bump? During iteration the consuming project resolves your card via file:. That ref has range * and re-resolves on every write. You don't bump the version until you're cutting a release that others should pin against. See Scenario B.

Scenario B · Cutting a release of your card

Goal: turn the current source contents into a versioned, distributable release. Consumers pull the new version through card update or by re-applying.

Commands

# 1. Stamp a new version on the source. Choose:
#    patch — bug fix, no structural change
#    minor — adds skills/MCPs or relaxes constraints
#    major — removes / renames skills, breaking change
$ drwn card source set @scope/name --version 1.2.0

# 2. Publish locally (creates a tagged commit in the bare repo + extracted snapshot).
$ drwn card publish @scope/name

# 3. (Optional) Push to your Git remote.
$ drwn card push @scope/name

# 4. (Optional) Catalog publish if you maintain a catalog.
$ drwn card catalog publish @scope/name@1.2.0 --catalog <target>

What's happening under the hood

StepLayer touchedEffect
source set --versionSourceMutates sources/@scope/name/card.json.version. Strict semver required.
card publishSource → StoreStages source dir, commits to ~/.agents/drwn/cards/@scope/name.git/ on main, tags v1.2.0, extracts to ~/.agents/drwn/extracted/<sha256>/. Integrity hash computed over the extracted tree. Guardrail: rejects if your version bump doesn't match the structural classification.
card pushStore → Remotegit push origin refs/heads/main --tags. Local [drwn] config never travels.

Consumer side

# Consumers see the new version once they refresh their lock.
$ cd <their-project>
$ drwn card outdated            # report-only, exits 0
$ drwn card update               # re-resolves every ref, writes new card.lock
$ drwn write --dry-run
$ drwn write
Publish guardrail card publish classifies the structural diff vs the previous version and rejects bumps that understate the change (e.g. adding a skill but only bumping patch). Use --force-bump-mismatch only if you intentionally bypass — it almost always indicates the version bump was wrong.

Scenario C · Creating a brand-new card from scratch

Goal: stand up a new card source with one or more skills, then publish it.

Commands

# 1. Create the empty source. Scope is auto-derived from `gh` / git config
#    on first use and saved to machine.json.
$ drwn card new @scope/name --no-git

# 2. Set the description (good hygiene; surfaces in `card show`).
$ drwn card source set @scope/name \
    --description "What this card bundles and who it's for"

# 3. Stage your skill content. Each skill is a directory with SKILL.md
#    plus any auxiliary files (scripts, fixtures, prebuilt JS bundles).
$ mkdir -p /tmp/staging/my-skill
$ vim /tmp/staging/my-skill/SKILL.md   # frontmatter: name + description

# 4. Add into the source (one call per skill).
$ drwn card source add-skill @scope/name my-skill \
    --from /tmp/staging/my-skill

# 5. (Optional) Add MCP servers to the card.
$ drwn card source add-mcp @scope/name <server-id>

# 6. Validate.
$ drwn card source doctor @scope/name --json

# 7. Test against a scratch project using a `file:` ref (no publish yet).
$ mkdir -p /tmp/test-project && cd /tmp/test-project
$ drwn init --non-interactive
$ drwn card apply file:$HOME/.agents/drwn/sources/@scope/name
$ drwn write --dry-run
$ drwn write

# 8. Smoke test (open Claude in /tmp/test-project, try invoking a skill).
# 9. Publish + push when ready — see Scenario B.

What's happening under the hood

StepLayer touchedEffect
card newMachine + SourceCreates sources/@scope/name/ with an empty card.json (version 1.0.0, empty skills.include). Without --no-git, initializes a working git repo inside the source.
source set --descriptionSourceMutates card.json.description. Also accepts --license, --harness-min-version, --stability, --last-validated-with, --test-status-badge.
source add-skillSourcecp -r the --from dir into sources/@scope/name/skills/<skill>/. Appends to card.json.skills.include. SKILL.md is required; everything else travels.
source add-mcpSourceAdds mcp-servers/<id>.json and records it under card.json.servers.
source doctorSource (read-only)Reports orphaned bundled skills/MCPs, missing manifest entries, package.json divergence, manifest schema violations.
card apply file:...ProjectResolves the source directly (no publish needed), validates its card.json + skill dirs, writes config.json + card.lock with the file ref.
Auxiliary files Skills can ship .ts, .sh, prebuilt JS bundles, fixtures — anything under the staged directory. The whole skill dir is symlinked into .claude/skills/<name>/ at write time, so the body can reference siblings via stable paths like node .claude/skills/my-skill/dist/index.js.

Scenario D · Capturing an existing project as a card

Goal: snapshot a project's effective harness (its currently active skills and MCPs) into a self-contained, sharable card source.

Commands

$ cd <project-with-harness>
$ drwn card new @scope/project-harness --from-project . --no-git

What's happening under the hood

StepLayer touchedEffect
card new --from-projectProject → SourceReads the project's effective skill set via the layered resolver. Copies each selected skill into sources/@scope/project-harness/skills/<name>/. Writes a card.json with version 0.1.0 by convention; you bump it up to 1.0.0 before publishing.

After capture you treat it like any other source: edit, doctor, apply to a scratch project, publish.

Scenario E · Switching the whole card set on a project

Goal: replace whatever cards are applied with a different set. Useful when changing context (research project → production project), or when consolidating onto a new card.

Commands

$ cd <project>

# 1. See what's currently applied.
$ drwn card status --json

# 2. Replace the set in one shot. Any number of refs — this is the whole desired state.
$ drwn apply @scope/card-a@^1.0.0 @scope/card-b@^2.0.0

# 3. Preview downstream change.
$ drwn write --dry-run --json

# 4. Apply downstream.
$ drwn write

What's happening under the hood

StepLayer touchedEffect
drwn applyProjectReplaces config.json.cards with the provided list. Resolves each ref against the local store / file source. Rewrites card.lock in one transaction. Old cards drop out of the array.
drwn writeProject → DownstreamRecomputes the effective skill + MCP set. Stale drwn-managed symlinks (from the previously applied set) are removed; user-owned files with the same name are preserved with a warning.
apply is destructive at the config level The previous cards array is overwritten in full. There is no --dry-run on card-mutation commands today; the safety net is drwn write --dry-run showing the downstream diff. If you only want to add one card, use drwn card add (Scenario F).

Scenario F · Adding a card alongside others

Goal: layer one more card on top of what's already applied.

Commands

$ cd <project>

# 1. Append a single card.
$ drwn card add @scope/extra-card@^1.0.0

# 2. Materialize.
$ drwn write --dry-run
$ drwn write

What's happening under the hood

StepLayer touchedEffect
card addProjectAppends one ref to config.json.cards. Refreshes card.lock. Duplicate card names are rejected — use card pin or card update if it's already there.
drwn writeProject → DownstreamLayers the new card on top of existing materialization. Skill / MCP collisions resolve per the rules in Conflict resolution.

Scenario G · Removing or detaching cards

Two distinct operations: remove one card by name, or detach the project from all cards.

Commands

# Remove a single card by name.
$ drwn card remove @scope/card-a
$ drwn write

# Detach the project from all cards (overlay preserved).
$ drwn card detach
$ drwn write

What's happening under the hood

StepLayer touchedEffect
card removeProjectDrops the named entry from config.json.cards. Rejects unknown names so scripts catch stale assumptions.
card detachProjectClears the cards array but preserves explicit project overlay (direct skills, servers, extensions, targets). Useful when transitioning to non-card configuration without losing per-project additions.
drwn writeProject → DownstreamRemoves drwn-managed symlinks that the dropped cards used to satisfy. Skills that overlapped with another card or with machine defaults stay in place.

Scenario H · Version mechanics

Three commands cover the lifecycle of a card's version constraint on a project.

Inspect what's outdated

$ drwn card outdated          # human-readable
$ drwn card outdated --json   # machine-readable
$ drwn card outdated --check  # exit non-zero if updates exist (CI-friendly)

Refreshes the lockfile, compares locked versions to the highest local version that satisfies each ref's range, and reports the deltas.

Re-resolve all refs against the current store

$ drwn card update
$ drwn write

Re-resolves every ref in config.json and writes a fresh card.lock. Use this after publishing newer local versions that satisfy your configured ranges, or after fetching from a git remote.

Pin to an exact version

$ drwn card pin @scope/name@1.2.3
$ drwn write

Replaces the version constraint for one card. Locks reproducible behavior independent of future publishes. If the card isn't already in the project, pin adds it.

Four inspection lenses, no mutation.

CommandAnswers

drwn status --why <name>Which layer is providing <name> and why? Card vs machine default vs project overlay.

drwn card statusJust the cards: specs, locked versions, integrity, paths.

Conflict resolution rules

When the same skill name is available from multiple sources, drwn resolves in a fixed order (card-skill-resolver.ts:31–65):

  1. Cards, in array order. The first card in config.json.cards that declares the skill wins. Order matters.
  2. Machine defaults (user-default layer). The skill library set via drwn library defaults add skill <name>.
  3. Missing. Reported as a resolution failure during drwn write.

Practical implications:

  • To override a machine default for one project, ship a card that declares the same skill name.
  • To pick a winner between two cards that both declare morning, reorder them via drwn apply <winner> <loser> or rebuild the array with card remove / card add.
  • drwn status --why morning tells you which layer is actually providing the skill in the current state. Run it before assuming.

MCP servers and extensions have analogous layering. For per-tool detail run drwn status --explain.
