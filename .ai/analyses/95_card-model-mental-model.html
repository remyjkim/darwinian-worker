1. The model in one idea

A card — and every skill, MCP, or hook inside it — is never “just a file.” At any moment it occupies a state, and every CLI subcommand is a transition between states. Understanding the model means knowing (a) the states, (b) which are reversible, (c) which are forbidden, and (d) the one command that performs each legal move.

Two rules cut across everything:

  1. Two planes. Content (a published version) is immutable — it is never edited, only superseded. Metadata about content (deprecation, successor) is mutable and rides a separate plane that travels with the card.
  2. Availability ≠ activation. A card being in your registry (machine-wide) is a different state from it being used by a project (per-project). No command silently crosses that line.
immutable content mutable metadata activation / project working / mutable draft forbidden / refused

6. Scenarios the model accounts for

Each is a path through the state machines above. The point of the model is that every one of these has a single, unambiguous command sequence — and that the wrong moves are structurally prevented.

A. Fix a typo in a published skill

Canonical (edit) → Bundled (sync) → Published (new version) → consumers: Outdated → Applied
edit Canonicalthe upstream repo — never the published version
→
card source syncrefresh the Draft’s bundled copy
→
card release --bump patchpublishes v+1 (old version stays frozen)
→
consumer: card update → drwn writepicks up v+1 within its range

B. Develop a card against your live working tree

Applied → Linked → Materialized (loop) → Applied
card link @me/x file:./xoverride → config.local.json (gitignored)
→
edit + drwn write --watchsee changes live; no publish, no path leak
→
card unlinkrestore the published pin
→
card releasewhen the edits are good, publish them

Why not card apply file:? Because that rewrites card.lock (shared) with your absolute path and destroys the pin. Linked state is machine-local and restorable — that’s the distinction the model draws.

8. Forbidden transitions (what the model refuses)

A mental model is defined as much by its illegal moves as its legal ones. These are structural, not advisory — the tooling prevents them:

Edit a Published version. Immutable by construction (content-addressed). The only way “forward” from Published is a new publish. → protects reproducibility & team pins.
Make a Materialized copy authoritative. Hand-edits put it in Drifted; drwn write refuses rather than clobber, and points you at Canonical. → the copy is a projection, never a source.
Leak a dev override to teammates. Linked state lives only in config.local.json (gitignored); it can never enter config.json or card.lock. → shareable intent stays clean.
Clobber metadata across machines. The metadata plane is union-merged, never force-pushed. → two machines’ deprecations coexist.
Activate machine-wide by accident. A write with no project above cwd requires explicit --scope machine. → availability never silently becomes activation.
Trust a cross-scope successor blindly. A successor pointer to a different scope is only a suggestion until you confirm or a catalog corroborates. → renames can’t be weaponized.

Updatescard source sync (source-side), outdated+update (project-side), projects update --all (fleet), deprecate (metadata)updates happen at the layer that owns the truth — upstream for skills, lock for pins, metadata plane for lifecycle

The through-line: content flows one direction and is immutable; metadata rides a parallel mutable plane; availability and activation are separate states; and the dev loop is an explicit, machine-local override — never a rewrite of shared truth. Every command is the named door between two of these states, and the doors that would corrupt the model simply aren’t built.
