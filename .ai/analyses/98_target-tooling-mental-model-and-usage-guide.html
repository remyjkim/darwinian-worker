1. The whole model in one idea

There are three places a card lives, and one machine-local scratch layer. Content flows downward through them, and nothing crosses a boundary without an explicit command:

① SOURCE — where you write
Editable card drafts. Managed centrally in your home store, or cloned as a shared org monorepo at $CARDS_SOURCE_PATH. Each source has its own git remote. — the only intended edit point.
▼  card publish
② STORE — the immutable ledger (machine-wide)
~/.agents/drwn/ holds every published version, content-addressed (cards/*.git + extracted/<treeSha>). Shared by every project and worktree. — availability, never edited.
▼  card apply (pin)  +  drwn write (materialize)
③ PROJECT — committed activation (per branch / worktree)
<project>/.agents/drwn/: config.json (intent) + card.lock (pins) + vendor/@scope/name/<sha>/ (content, CoW-cloned (reflinked) from the store, committed as real files). Projection surfaces (.claude/skills/, .cursor/, project .mcp.json) are generated from vendor and gitignored; merge surfaces (machine ~/.claude.json, .codex/config.toml, .claude/settings.json hooks) are field-merged (§3.5). — usage. This is branch content, so it diverges per worktree for free.
◆ LOCAL OVERLAY — machine-local, gitignored
config.local.json (personal card activation + dev links) + card.lock.local (their pins). Layered on top of ③ at resolve time; never enters the committed lane.
source (editable) store (immutable, machine-wide) project (committed, per-branch) local overlay (gitignored) forbidden / refused
The one sentence to remember: the store is a shared cache of every version; a project commits exactly the versions it pins, as self-contained reflinked (CoW-cloned) files, so a teammate can git clone the repo and run drwn write offline — no store, no network, no author's machine.

2. The three things this must let you do

#You want to…How the model delivers it
1Keep sources in one home place, but commit materialized cards into a project repo Sources in tier ①; materialization vendors content into tier ③ as committable real files (no store symlinks).
2Materialize different versions / sets of cards across git worktrees or branches config.json+card.lock+vendor/ are committed branch content → each worktree materializes its own set. The shared store holds all versions, so no contention.
3Keep git remotes for sources and know when to version-up or edit Sources = a shared org monorepo with remotes; drwn up/outdated notify pull-on-command; editing is fork-first. V1 caveat: version-up ships now; distributable deprecation / successor notification waits on catalog reflection (V2).

3. Two ways a card materializes — and how the mode is chosen

Every card resolves to one of two materialization modes. Linking is always a local overlay — it never changes what is committed.

ModeWhat happensCommitted?For
vendored Pinned <sha> reflinked (CoW-cloned) from the store into vendor/; surfaces generated from it Yes — self-contained bytesConsumers; anything you commit
linked Content resolved live from the source working tree under $CARDS_SOURCE_PATH; edits reflected on next drwn write No — machine-local onlyAuthors developing a card (the dev loop)

Resolution precedence, highest wins (§8 of analysis 97 — explicit committed config outranks per-card auto):

1. drwn dev @scope/x → linked # explicit invocation override 2. project config materialization: "…" → as set # explicit, committed — "vendored" wins unconditionally; # "linked" links if source present, else vendored+warn 3. CARDS_SOURCE_PATH set & source present → linked # per-card auto (only when no explicit setting) 4. otherwise (default, per-project): → vendored # consumer, or source-absent
drwn write and drwn status print the resolved mode per card and why (@you/x → linked from ~/org-cards/x; @team/y@2.1 → vendored), so the behavior is never a surprise — to a human or to an agent operator. A committed materialization: "vendored" is never silently overridden by a stray CARDS_SOURCE_PATH.

3.5 Two kinds of agent surface — not everything is a gitignored projection

The files agent tools read split by who owns them. Treating them uniformly loses user data (a purely-regenerated field-merged config drops the user's own entries), so:

KindExamplesOwnershipCommitted?Guarantee
Projection.claude/skills/<name>/, .cursor/, project .mcp.json, .cursor/mcp.jsondrwn owns the whole file/dirno (gitignored, regenerated)byte-identical, offline
Mergemachine ~/.claude.json, .codex/config.toml, .claude/settings.json (hooks)drwn owns only its fields; user entries preservedfield-merged in placedeterministic per drwn-owned field
Project .mcp.json and .cursor/mcp.json are whole-file projections drwn owns outright (matching the shipped code) — do not confuse them with the machine-level ~/.claude.json merge surface. Which fields drwn owns on a genuine merge surface is recomputed from the pinned cards on every write — so on a teammate's fresh checkout (no local write-record) drwn still knows exactly which entries are its own.

5. Usage walkthroughs

A. Consume a card into a project (the 10-minute path)

discoverdrwn search card
→
into storedrwn card clone <url>
→
pindrwn card apply @t/y@^2.0
→
vendor + surfacesdrwn write

drwn use @team/y@^2.0 fuses the last three into one command. Result: config.json, card.lock, and vendor/<sha>/ are ready to git commit; .claude/ etc. are generated and gitignored.

B. A teammate checks out the repo (offline, no store)

clone repogit clone …
→
vendor bytes already present(committed)
→
regenerate surfacesdrwn write

No network, no card remote, no populated store required — the vendored trees are the content. This is the payoff of committing tier ③.

C. Develop a card against your live tree (the dev loop)

edit source~/org-cards/x/…
→
turn on linkdrwn dev @you/x
→
see it livedrwn write --watch
→
ship itdrwn release @you/x
→
restore pindrwn dev --off

While linked, edits to the source materialize immediately — no publish. The link lives only in config.local.json; your committed card.lock/vendor/ are untouched, so nothing leaks to teammates.

D. Different cards per branch / worktree (requirement 2)

worktree A (branch main)git worktree add ../A
→
pins X@1.0, Y@2.0drwn write
|
worktree B (branch exp)git worktree add ../B
→
pins X@2.0, Z@1.0drwn write

Because config.json/card.lock/vendor/ are branch content, each worktree simply materializes what its branch pins. The shared store already holds X@1.0 and X@2.0, so there is zero contention. (Explicit worktree helpers and worktree-aware GC are V2; the divergence itself works today.)

E. Update to a newer version (requirement 3, pull-on-command)

check remotesdrwn card outdated --fetch
→
re-pin + re-vendor + regen, whole projectdrwn up
→
review the diffgit diff vendor/

A vendored version bump is an add-tree / remove-tree change in vendor/ — reviewable in the PR. You are never surprised: nothing updates until you run up/update.

F. Edit someone else's card (fork-first)

fork the sourcedrwn card fork @team/y
→
edit under your scope@you/y
→
publish + pushdrwn release @you/y
→
adopt itdrwn use @you/y

Editing happens in the source lane (your scope / the org monorepo), fully decoupled from any project repo — so card maintenance never collides with project development.

6. Forbidden moves (the tooling refuses these)

Edit a materialized surface. Hand-editing .claude/skills/x/SKILL.md puts it in Drifted; drwn write refuses and prints a signpost to the source edit point (git+…#skills/x when the card's upstream is known, else drwn card fork → edit → publish → update). Your edit is preserved, not clobbered. The surface is a projection, and vendor/ is an immutable cache — neither is an edit target.
Edit a published version. Immutable by construction — fix by publishing the next version.
Leak a dev link or a personal card to teammates. Links and local activation live only in config.local.json / card.lock.local (gitignored); they can never enter config.json/card.lock.
Commit a store symlink. Not possible — vendored content is real files reflinked (CoW-cloned) from the store, so a checkout on another machine gets bytes, not a dangling link.
Poison the store by editing vendor/. Also not possible — the reflink is an independent inode, so an in-place edit to vendor/… can never write through to ~/.agents/drwn/. And because vendor/ is an immutable cache, drwn write re-verifies against the content manifest and repairs the pinned tree from the store (overwriting the edit) — it is not an accepted edit point.
