# Worker Instructions Projection (AGENTS.md) — Target Architecture

Design for incorporating the canonical-system-prompt setup from
`.ai/analyses/123_system_prompt_architecture_analysis.md` into the drwn CLI: project the
active Worker's composed instructions into a repository-root `AGENTS.md` managed block,
with thin per-tool adapters only where a harness does not consume `AGENTS.md` natively.

**Inputs:**
- `.ai/analyses/123_system_prompt_architecture_analysis.md` — the cross-agent research:
  root `AGENTS.md` as the canonical instruction file; `.claude/CLAUDE.md` one-line import
  adapter; native consumption by Codex, OpenCode, and Cursor; drift sentinels; "prompts
  are not security enforcement."
- `.ai/analyses/100_*card_instructions_projection_gap_analysis.md` — the missing last hop:
  instructions are validated, locked, and materialized but no local harness surface reads
  them.
- `.ai/analyses/101_*card_spine_delivery_design_options_analysis.md` — the delivery-context
  option families this design resolves for the main-agent context.
- `.ai/analyses/122_*opencode_target_support_target_architecture.md` §D8 — the per-target
  instruction-surface question this design dissolves.
- Codebase investigation (2026-07-22), file:line against current `main` (`e4d81c7`).

## 2. What analysis 123 changes about the instructions-delivery decision (I24: "spine delivery")

Docs 100/101 left open *which mechanism* should carry instructions into local harnesses
(three option families: harness-native sub-worker surfaces, main-conversation context, pull
conventions). The 123 research resolves the main-conversation family concretely:

- **Every current drwn target consumes root `AGENTS.md`**: Codex natively (layered chain),
  OpenCode natively (preferred over its CLAUDE.md fallback), Cursor natively (root +
  nested), and Claude Code via a one-line `.claude/CLAUDE.md` import adapter.
- This **dissolves design-122 D8's per-target instruction surface**: there is no
  per-target file matrix to design. One canonical projection + one adapter covers all
  four targets, and a future gemini target needs only a `.gemini/settings.json` managed
  field (`context.fileName`), not a new content path.
- 123's constraints adopted as design invariants: never replace vendor system prompts;
  root file only for mandatory content (nested discovery semantics differ per tool); a
  sentinel (`Instruction-ID`/content hash) for drift verification; prompts are guidance —
  enforcement stays in hooks/permissions (which drwn already projects per-runtime).

Doc 101's Families 1 (sub-worker surfaces) and 3 (the instructions-skill convention,
historically "spine-skill") remain open for
their own contexts; this design claims only the main-agent context.

## 3. Design

### D1 — Project a drwn-managed block inside root `AGENTS.md`

`AGENTS.md` is conventionally user-authored and committed — drwn must never own the whole
file (same reasoning as `opencode.json` in design 122 D4/D9). Instead drwn owns a
marker-delimited block, exactly the pattern `git-hygiene.ts` already implements for
`.gitignore` (`DRWN_GITIGNORE_MARKER` + `extractDrwnBlock`/`renderDrwnBlock`,
`git-hygiene.ts:9,40-58`):

```md
<!-- drwn:instructions:start -->
<!-- Generated by drwn from the active Worker closure. Do not edit this section.
     Instruction-ID: <workerName>@<version>  Content-Hash: sha256-… -->
…composed instructions (the same bytes as .agents/drwn/generated/instructions.md)…
<!-- drwn:instructions:end -->
```

- Everything outside the markers is user-owned and passes through byte-for-byte.
- A missing file is seeded with the block only (user section grows organically).
- HTML comments render invisibly and survive every consumer in 123's matrix.
- Two hash domains, deliberately distinct (072326, review01 R1-F09): the header
  `Content-Hash` is the SHA-256 of the composed instruction bytes only — the drift
  sentinel 123 recommends. The write-record ownership hash covers the exact rendered
  managed block (markers and header included), so cleanup can prove the whole block is
  unmodified. Conflating them would let a header-only tamper defeat safe cleanup.

**Ownership model:** `managed-fields`-style entry with a single field (the block), path
`AGENTS.md`, new surface value `"instructions"` in `ProjectionSurface`
(`write-record.ts:32`), `target` unset (target-agnostic, like `worker`). Drift on the
block throws with the standard `--force` remediation; foreign content (user sections) is
never touched; cleanup removes only an unmodified block.

### D2 — Claude adapter: `.claude/CLAUDE.md` import, guarded

Per 123, the adapter is the literal line `@../AGENTS.md`, placed at `.claude/CLAUDE.md`
(not root `CLAUDE.md`, which Cursor would double-read).

- **File absent** → drwn writes the one-line adapter, recorded `managed-content`.
- **File present and not drwn-owned** → never touch it silently. Doctor emits an
  advisory ("`.claude/CLAUDE.md` exists but does not import `AGENTS.md`"), and an
  explicit fix verb (`drwn doctor --fix-claude-adapter`-style command) applies the
  import line as a drwn-marked block on deliberate request — recorded as owned, cleanly
  removable. Never an implicit edit; this extends the cursor-`hooks.json` foreign-file
  precedent with a consented escape hatch. (Decided 072226, doc 125 B-Q2.)
- Cleanup removes the adapter only when its content hash matches (standard
  managed-content semantics).

Codex, OpenCode, and Cursor need no adapter (native root `AGENTS.md`). Gemini's
`.gemini/settings.json` adapter is deferred until a gemini target exists.

### D3 — Pipeline placement and gating

New `syncInstructions(state, previousManagedPaths)` step in the write pipeline beside
`syncWorkers` (`sync.ts` main flow).

**Contribution contract (decided 072326, review01 R1-F01):** a card contributes
instructions iff it declares them deliberately — card `manifest.instructions` or
Blueprint `identity.instructions`. The current composer's fallback to bundled skill
bodies (`sync-worker.ts:95-140`) is **retired for this path**: skills are model-facing
procedures, not persistent behavioral contributions, and a consent grant over
"instructions" must never silently cover skill-derived text (the org-layer grammar
draws the same line — doc 08 §7.7 separates implementation bindings from contribution
consents, and its validation rejects bindings that confuse the two). One canonical
contribution resolver feeds consent checks, the content digest, doctor staleness, and
the projected block, so their bytes cannot disagree. The byte-identity invariant is
scoped to this consented composition (review01 R1-F02); the worker-generator's
internal artifact keeps its current composition and is not the projection source.

- **Project scope only** in phase 1. Machine scope (`~/.claude/CLAUDE.md`,
  `~/.codex/AGENTS.md`, `~/.config/opencode/AGENTS.md`) is a different trust surface —
  deferred, noted as a later phase.
- **Runs when the active Worker closure yields non-empty instructions**; no active worker
  or empty composition → no block written, prior-owned block cleaned up.
- **Consent gate (decided 072226, doc 125 B-Q1):** projection requires instruction
  consent per card through the card-trust surface — `drwn card trust <card>
  --instructions`, semver-ranged and recorded in `card.lock` exactly like hook consent.
  Consent is **always explicit** — no first-party auto-grant in phase 1 (decided
  072326): `CardOrigin` cannot prove local authorship (review01 R1-F03 — a third-party
  card resolved through the local store would silently auto-consent), and the org-layer
  contract makes every consent an explicit, ratifier-attributed record (doc 08
  `ContributionConsentSpec`). An unconsented card's contribution is excluded from the composed block with
  a warning (per-card exclusion, mirroring hook-policy skips); `drwn write --strict`
  fails instead, giving CI enforcement. This supersedes the earlier on-by-default
  recommendation: a projected instruction block from a third-party card is an injection
  channel, and ungated is not an answer (I24 criterion 4).
- `--mcp-only`/`--skills-only`/`--target=<t>` narrowing: instructions are target-agnostic;
  they materialize on full writes only (skipped under any narrowing flag), mirroring how
  workers materialize today.

### D4 — Hygiene, watch, diagnostics

- **git-hygiene:** never ignore `AGENTS.md` or `.claude/CLAUDE.md` — both are
  conventionally committed. No `PROJECTION_SURFACE_ENTRIES` additions. (Deliberate
  divergence from the skills/mcp surfaces, same reasoning as `opencode.json`.)
- **write-watch:** add both paths to the watch set so `--watch` re-syncs on drift, or at
  minimum exclude nothing (they are user-editable files drwn merges into).
- **doctor:** three checks — (1) block present + hash matches the composed instructions
  (staleness), (2) `.claude/CLAUDE.md` adapter present or advisory, (3) sentinel
  `Instruction-ID` matches the active worker@version. These are the CLI-native version of
  123's CI verification script.
- **ambient-capabilities:** observe root `AGENTS.md` (present/absent, drwn block
  present/absent) so `drwn status` can report instruction delivery state per project.

### D5 — What this deliberately does not do

- No literal system-prompt replacement (`--system-prompt-file`, `GEMINI_SYSTEM_MD`,
  `model_instructions_file`) — 123's core warning; drwn stays on the context-file layer.
- No nested `AGENTS.md` management — root only (nested semantics differ per tool).
- No per-target instruction variants — one canonical document; per-target behavior
  belongs in hooks/permissions, which drwn already projects.
- No org-policy sync (123 §organization-wide) — that is a catalog/profile concern for a
  later phase; the block header's `Instruction-ID` leaves room for it.
- No second composer — a positive constraint on future work: any sub-worker content
  type (the A+O×N workstream) must feed the same composition that produces the worker
  instructions; two composers guarantee drift (I24 criterion 5; doc 125 §A4).

## 4. Implementation sketch (phase 1)

1. Prep: extract the `.gitignore` block helpers into a shared
   `managed-block.ts` (marker-parameterized `extractBlock`/`renderBlock`) — pure
   refactor, git-hygiene keeps behavior.
2. `write-record.ts`: `ProjectionSurface` gains `"instructions"`; ownership rule:
   `instructions` → target undefined. Machine-scope records continue to reject it.
3. `cli/core/sync-instructions.ts` (new): compose (reuse the sync-worker composer),
   gate per card on instruction consent (reuse the hook-consent pattern:
   `isInstructionConsentValid`, strict-mode failure), render block with
   `Instruction-ID` + content hash, merge into `AGENTS.md`, write `.claude/CLAUDE.md`
   adapter per D2, record ownership, cleanup on empty/none.
3a. Trust surface: `drwn card trust <card> --instructions` / `untrust`, recorded in
   `card.lock` beside hook consent; always explicit (no auto-grant).
3b. Authoring (decided 072226, doc 125 B-Q4): `--instructions` on `drwn card source
   set` (text or path), and fix the `project-worker-v1.md:108` contract line to
   describe the now-real projection.
4. `sync.ts`: invoke beside `syncWorkers` under full-write conditions; thread
   `previousRecord`.
5. Doctor + ambient observation per D4.
6. Docs: cli-quickref, per-project-config, how-apply-works — in `docs-docusaurus/`,
   the active docs tree (`docs-astro/` is deprecated per its `DEPRECATED.md`);
   CHANGELOG.
7. Tests (patterns already established this cycle): block merge preserves user sections;
   foreign `.claude/CLAUDE.md` untouched + advisory + fix verb; consent gate (unconsented
   card excluded + warning; `--strict` fails); drift throw +
   `--force`; cleanup on worker removal; adapter lifecycle; doctor staleness; e2e `drwn
   write` in a project with an instruction-bearing card asserting both files; binary
   smoke against real
   `opencode debug` (AGENTS.md pickup is directly observable via
   `opencode run`/context) and cursor-agent where feasible.

Estimated shape: ~12 bite-sized TDD tasks; no schema migrations; no breaking changes.

## 5. Decisions (approved 072226 — full option analysis in doc 125 Part B)

- **Q1 (D3): consent-gated activation** via the card-trust surface (`card trust
  --instructions`), `--strict` CI enforcement. Supersedes this
  doc's original on-by-default recommendation (overturned by I24 criterion 4). The
  first-party auto-grant element was removed 072326 — see D3's consent-gate contract
  (review01 R1-F03; doc 08 requires ratifier-attributed explicit consent).
- **Q2 (D2): advisory plus explicit fix verb** for a foreign `.claude/CLAUDE.md`; never
  an implicit edit.
- **Q3 (D1): full composed content in the block** — pointers would break
  codex/opencode/cursor and dangle for collaborators (generated dir is gitignored).
- **Q4: authoring folded into phase 1** — `card source set --instructions` plus the
  `project-worker-v1.md:108` contract-line fix (I24 lean layer 1: "immediately and
  regardless").
- **Mechanism (doc 125 B-Q6): C only for phase 1** — session-start hook injection (D)
  stays shelved as a possible claude/codex complement; doctor staleness checks cover the
  deleted-block case.
- **Scope (doc 125 B-Q5): main worker instructions only, cross-target** — sub-worker
  instructions proceed as their own workstream (A+O×N, four targets); this doc's only
  obligation is the shared-composer invariant (D5).

Amendments decided 072326 (sources: `cl0024_review01` execution-readiness review;
`darwinian-org/.ai/analyses/08_architect_organization_provisioning_blueprint_target_architecture.md`):

- **Contribution contract:** explicit instructions only; the bundled-skill fallback is
  retired for the projection path; one canonical contribution resolver (D3,
  review01 R1-F01/R1-F02).
- **No first-party auto-grant** in phase 1; consent is always an explicit record (D3,
  review01 R1-F03).
- **Dual hash domains:** content digest vs managed-block ownership hash (D1,
  review01 R1-F09).
- **Diagnostics contract** (doctor/status issue codes, JSON fields, severities, exit
  behavior) is a named plan-level design obligation, not an afterthought
  (review01 R1-F14).

## 7. Upstream interface (org provisioning layer)

The org side compiles worker inputs from the accepted
`OrganizationProvisioningBlueprintV1`
(`darwinian-org/.ai/analyses/08_architect_organization_provisioning_blueprint_target_architecture.md`
— direction approved by the owner; exact schema pending acceptance). Alignment
obligations on this design:

- A drwn instruction-consent record is the worker-local materialization of an org
  `ContributionConsentSpec` (`contributionKind: 'instructions'`, org projection surface
  `'worker_instructions'`): `ratifierRef` maps to the grant's `grantedBy`, the
  consented semver range to `consentedRange`, and the content digest to the trust
  acknowledgement digest. Because every org consent is explicit and
  ratifier-attributed, the drwn surface needs no implicit grant path — reinforcing the
  no-auto-grant decision.
- Card identity arrives pinned (`ArtifactPinSpec`: name, version, integrity, origin)
  in the `OrgWorkerBundleV1` projection; consent ranges express future compatibility
  and never replace the exact pinned digest for one accepted revision.
- drwn remains the sole harness materializer (doc 08 §6.1: never authored by
  Architect); the bundle carries no harness file bodies or inline instruction content.
- Doc 08 §19.6.4 confirms this workstream's scope verbatim: Darwinian Worker
  instruction projection, materialization, consent, diagnostics, and worker-local
  receipts.

## 8. Test intent (GATE 1)

What the implementation must prove, independent of task breakdown (the GATE 2 plan
carries the exact TDD sequence):

- **Managed block safety:** bytes outside the markers preserved exactly (newline
  style, final-newline state); malformed or ambiguous markers (missing end, duplicate,
  reversed, marker-like user text) fail closed without mutation; CRLF and empty-file
  cases pinned.
- **Contribution resolver parity:** consent filtering, content digest, doctor
  staleness, and the projected block all derive from the same resolver — a fixture
  with one consented and one unconsented contributing card proves exact expected
  bytes for both the block and the exclusion warning.
- **Consent lifecycle:** grant/untrust round-trip in `card.lock`; carry-forward across
  in-range card updates; drop-with-warning outside the range; `card outdated`
  readout; digest re-acknowledgement on content change.
- **Hash domains:** tampering with body, header, and markers independently — content
  hash catches body drift; ownership hash catches all three; cleanup refuses when the
  full block bytes differ.
- **Adapter lifecycle:** absent → written and owned; foreign without import →
  untouched + advisory; foreign with valid import → satisfied, no write; owned then
  user-edited → drift handling; removal cleans only matching content.
- **Modes:** unconsented card warns on normal write and fails `drwn write --strict`;
  narrowing flags skip instructions; empty composition removes a previously owned
  block and nothing else.
- **End to end:** `drwn write` in a project with an instruction-bearing consented card
  produces both files, idempotent on bytes and mtime; live-session V1–V3 remain
  scheduled manual gates with operator evidence, never claimed from config
  inspection alone.
