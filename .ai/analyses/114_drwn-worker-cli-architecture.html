The Darwinian Worker CLI

Cards compose into a Blueprint, a project selects at most one Worker, and write projects declared intent.

first supported local contract | drwn 0.9.0 candidate | reviewed 2026-07-13
Public mental model: Cards are independently versioned capability packages. A Blueprint is a Card that composes plain Cards and governance. Each installed root is a possible Worker; exactly one root or none is selected. The selected root and its closure provide project behavior. Machine capabilities are separately declared ambient tooling, never implicit members of the project Worker.

1. Canonical Object Model

Workers are alternatives, not a stack. Capability composition happens once, inside a Blueprint. A plain Card root is a valid one-Card Worker. Nested Blueprints are rejected.

2. Authority And Data Flow

Card sources / catalogs
  -> immutable Card store and standalone machine inventory
  -> project requirements + explicit machine intent
  -> immutable project lock
  -> effective selected Worker state
  -> generated Worker aggregates
  -> downstream project or user-home projections

The flow is one-way. Generated Worker directories, downstream tool files, user-home skill folders, and machine profiles never become project requirements. drwn write does not select a Worker, install a missing root, or repair declaration state.

3. First Supported Schemas

The prelaunch hard cut treats namespaced schema V1 records as the first public contract. Prototype readers, migration adapters, warning aliases, reconstruction heuristics, and dual writes are intentionally absent.

Project Config

{
  "schema": "drwn.project-config",
  "schemaVersion": 1,
  "workers": [
    "@team/reviewer-worker@^1.0.0",
    "@team/interviewer-worker@^1.0.0"
  ],
  "activeWorker": "@team/reviewer-worker"
}

workers contains ordered root requirements, not closure members. activeWorker is required and explicit. Project overlays for skills, MCP servers, hooks, extensions, targets, trust, and materialization are separate fields and do not alter the root graph.

Project Lock

{
  "schema": "drwn.project-lock",
  "schemaVersion": 1,
  "store": { "minDrwnVersion": "0.8.0" },
  "workerRoots": [
    {
      "name": "@team/reviewer-worker",
      "requested": "@team/reviewer-worker@^1.0.0",
      "kind": "blueprint",
      "members": ["@team/operator", "@team/notion"]
    }
  ],
  "cards": ["complete immutable Card entries omitted here"]
}

The lock stores every root and reachable Card exactly once, with explicit root-to-member edges. Root and Card order is deterministic; incompatible same-name resolutions and nested Blueprint members fail. Store/Git entries carry commit tree identity and integrity. A Mind-bearing graph raises the lock-wide floor to 0.9.0.

Generated Worker State

The CLI creates one aggregate directory per installed Worker root, not one directory per member Card. Multiple directories therefore represent alternative roots. Only the selected root's closure contributes active downstream project behavior. Member Cards remain independently pinned, vendored, verified, and attributed.

Fresh non-interactive setup creates explicit empty intent. Guided setup may offer an opt-out, immutable Recommended Darwinian Operator profile. A profile may project only approved machine-safe skills and MCP definitions. It cannot project Worker identity, instructions, hooks, permissions, governance, or Mind content.

Every managed path has an explicit capability surface and, when required, target owner. Project and machine records are scope-checked. Paths are contained relative paths; duplicate paths, unsupported ownership combinations, prototype records, and malformed hashes fail closed.

4. Project Resolution And Selection

  1. Declare: drwn add appends one root; drwn apply replaces roots.
  2. Resolve: recursively resolve each root, pin immutable Cards, and reject Blueprint recursion or version conflict.
  3. Select: drwn use <root> selects one installed root; drwn use --none selects none.
  4. Aggregate: compose the selected root followed by ordered members, then committed and local project overlays.
  5. Project: drwn write reconciles only declared output through the shared planner.

Adding the first root may explicitly select it. Several alternatives may remain installed and locked, but they contribute no active project capability. Selecting a Blueprint member directly is invalid because members are not roots.

5. Supported Command Grammar

NamespaceResponsibility
drwn initCreate namespaced project state and explicit machine intent; guided setup may offer the Recommended profile
drwn add | apply | pin | remove | update | useProject Worker requirements, immutable lock graph, and singular selection
drwn install | write | up | status | doctorHydration, pure projection, locked updates, explanation, and diagnostics
drwn add skill | add mcp | extensionsExplicit project capability overlays outside the Worker closure
drwn cardCard source authoring, validation, publication, catalogs, remotes, trust, and inspection
drwn workerBlueprint authoring and deployed Worker lifecycle
drwn worker mindOptional selected-Worker Mind provision, drift, sync, checkpoint, diagnosis, and retirement
drwn machine skill | mcp | inventoryStandalone machine inventory, explicit activation, references, portable transfer, and garbage collection
drwn devProject-local Card source linking and watch-write iteration

There is no library or broad store namespace. Standalone lifecycle and portable inventory live under machine; Card store behavior lives under card. Removed project mutations under drwn card, Worker stacking, and install --no-apply are ordinary unknown syntax, not aliases.

6. Pure Projection And Partial Writes

One shared projection planner drives real write, dry run, status, and doctor. It accepts resolved state and returns desired ownership, drift, collisions, and cleanup. Adapters render their target formats; they do not independently infer project intent.

InvocationReconcilesRetains
drwn writeAll selected project surfaces and enabled targetsNo stale selected ownership
drwn write --mcp-onlyMCP output for enabled targetsSkill, hook, and unselected-target ownership
drwn write --skills-onlyClaude/Codex skill outputMCP, hook, and unselected-target ownership
drwn write --target=<target>Selected target plus common generated Worker outputOther target ownership and bytes
drwn write --scope machineExplicit machine capabilities into user-home targetsProject state and project-owned output

Cleanup removes only previously owned, unchanged, now-stale output. User replacements and foreign content are preserved or block unsafe claims. A still-desired path changing representation is an ownership handoff, not permission to delete the desired result. --force cannot claim foreign paths or bypass target-native fatal MCP collisions.

7. Machine Inventory And Portability

~/.agents/drwn/
  sources/                 editable Card sources
  cards/                   bare Git repositories for immutable Card versions
  extracted/               verified immutable content trees
  skills/                  standalone immutable skill package versions
  mcp-servers/             standalone MCP inventory records
  machine.json             explicit profile, capability selections, and policy
  projects.json            registered project roots for reference safety
  credentials.json         authentication secrets; never portable inventory

Repository skills and bundled registry MCP definitions are discovery-only. Card-owned capability copies remain under Card lifecycle. Standalone installation is inactive until machine or explicit project intent references it. Package update/removal is package-scoped, reports affected IDs, and fails while referenced. Inventory mutation uses one global lock followed by machine and sorted project locks; ambiguous owners fail closed.

machine inventory export is the requirements-style canonical manifest. bundle adds verified payload bytes, verify compares source and target, and sync installs only missing inventory additively and inactive. Transfer excludes activation intent, credentials, Cards, project state, target output, history, and runtime state. Garbage collection is a dry-run plan by default and never removes current inventory merely because it has zero references.

8. Ambient Target Policy

Project evaluation does not inherit machine capabilities, but downstream tools may still expose user-home capabilities in project sessions. Status and doctor report that ambient visibility with provenance. Collision behavior is adapter-specific:

  • Claude whole-entry scope shadowing and transport replacement are reported without a universal block.
  • Codex table inheritance blocks incompatible effective transport combinations before mutation.
  • Cursor inheritance and project transport selection are warning-only under its native semantics.

Ambient definitions never become project declarations, and diagnostics never expose secret-bearing definition values.

9. Optional Worker Mind

Mind-capable selected closure =
  persona OR beliefs OR memory.observations OR memory.insights

/pool/observations/<date>/<HHmm>-<ulid>.jsonl
/pool/insights/<date>/<HHmm>-<ulid>.md

Mind operations validate the selected closure before network access. Persona-only and beliefs-only Workers are valid Minds; an empty memory object is not. BeginningDB stores a strict drwn.mind-index V1, semantic pool entries, and inode-backed views. The CLI provides no numbered-memory alias or migration path. raw_data is reserved and unsupported.

Forget uses explicit ?action=unplace; human-confirmed pool retirement alone uses delete_everywhere. See Analysis 113 for the complete engineering contract.

10. Deployment Boundary

drwn worker deploy <root-ref> --name reviewer
  1. Resolve one root and its complete closure.
  2. Verify Card identities, versions, integrity, and compatibility floor.
  3. Preserve the existing remote deployment payload contract.
  4. Poll readiness and cache only non-secret Worker/Mind coordinates.

The CLI resolves; the remote service materializes. Local V1 schema names do not leak into the existing deploy payload. Rollback changes the serving deployment while preserving remote Worker identity.
