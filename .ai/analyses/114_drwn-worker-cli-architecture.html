1The object model

Capability cards carry content (skills · hooks · MCP servers · mind seeds). Blueprints compose cards and carry governance. A worker is a deployed (or locally activated) blueprint instance — stateless by default. A mind is optional per-worker state: the live persona/beliefs/memory a worker gains when its stack composes mind-bearing cards and is provisioned. One composition engine, one lockfile, one materialization pipeline serve all of it.
The unit of mind, precisely: a worker is not a mind and a card is not a mind — a mind is state a worker has. Cardinality: worker (1) ──has── (0..1) mind · mind (1) ──seeded from── (1..n) cards · card (1) ──can seed── (0..n) minds. A worker has no persona, beliefs, or memory by default; it acquires a mind only when (1) at least one composed card carries persona/beliefs/memory sections and (2) worker mind provision has run. Naming fossil to know: the deploy API predates the Workers rename — its minds table and the mindId it returns are the worker's stable identity, minted for every deployed worker, mind or not. Identity always exists; the mind (the DB subtree) exists only once provisioned. And "mind card" means any mind-bearing card; the substrate is @darwinian/mind-tools (the mind-operating skills + conventions, no seeds) and the solo quickstart is @darwinian/mind-starter (voice + collaboration + the synced skills). The substrate is composed, never forked — minds are always defined by their own content card.

2Process model

Single-process, Bun-executed, Clipanion-based CLI. No daemon, no IPC, no cache outside the filesystem. Every invocation: read the on-disk surfaces → compute an effective state → report it (status, doctor) or materialize it (write). Network appears in exactly three places: git operations against card remotes, the deploy API (DAH-authenticated), and BeginningDB (mind operations). Everything else is local and deterministic — run write twice and the second run reports zero changes.

3Storage topology

# Machine-wide (the "store")
~/.agents/drwn/
├── sources/<scope>/<name>/        editable card sources (authoring home; publish reads here)
├── cards/<scope>/<name>.git       published cards: bare git repos, one tag per version (v1.2.0)
├── extracted/<treeSha>/           immutable checkouts of published trees (lock entries point here)
├── credentials.json               DAH tokens (keychain-encrypted)
├── mind-bindings.json             per-slug mind coordinates — never tokens (0.7.0)
├── machine.json · catalogs/ · generated/   machine config, fetched catalogs, machine-scope output

# Per project
<repo>/.agents/drwn/
├── config.json                    project config: cards[], activeWorkers[], servers/skills/targets
├── card.lock                      lockfileVersion 5: pinned versions + integrity + embedded manifests
└── generated/                     materialized worker bundles + workers.json (write output)

Two invariants: published versions are immutable (re-publishing an existing version throws — bump instead), and everything a card ships is integrity-hashed (whole-tree snapshot at publish; the lock records integrity + treeSha, verified on install).

4Card lifecycle

author    drwn card new @me/x --no-git · card source add-skill/add-hook/add-mcp/add-persona/add-belief
          card source doctor — structural issues (orphaned dirs, missing SKILL.md/PERSONA.md, …)
publish   drwn card publish @me/x
          validates declared content dirs → snapshots the WHOLE tree → semver-vs-diff classification
          (member-set / capability changes must match the bump) → tag v<version> in the bare repo
share     drwn card push @me/x [--remote-visibility …]     visibility gate: §10
          consumers fetch via store refs, git+https/github:/gitlab: refs, or file: (dev)
consume   drwn card apply <ref> [<ref>…]   multi-spec · resolves + locks + vendors
          drwn card add <ref>              exactly ONE spec
          card update / outdated / pin / diff / show / trust manage the pinned set
activate  drwn worker stack use <names…>   orders activeWorkers (§5)
emit      drwn write                        materialization (§6)

The lock (card.lock) is the project's ground truth: each entry pins name / requested / version / path / integrity / treeSha, embeds the full manifest, and lists resolved skills/hooks (+ mind sections at 0.7.0). Origins: store | git | file | npm.

5Composition & the effective state

One engine, three steps, all in-process:

  1. Resolve: each ref in config.cards[] resolves to a lock entry (store/git/file).
  2. Expand blueprints: for each kind: "blueprint" entry, its composedFrom members are resolved and spliced into the list after the blueprint (which remains, carrying governance). Members dedupe by name — first occurrence wins, including the pinned version. Blueprints may not compose blueprints (BLUEPRINT_RECURSION).
  3. Select: activeWorkers filters and orders the locked cards. If unset, all cards are active.
The no-expansion rule (the #1 operational surprise): selection is a plain name filter — neither worker stack use nor worker mind provision expands blueprints. A blueprint entry itself contributes zero skills and zero persona. Always list members explicitly:
drwn worker stack use @me/chief-worker @darwinian/mind-tools @me/chief

The resulting effective state (ordered active cards, their content roots, merged skills/servers/hooks, config overlays) is the single input to everything downstream: write, status --why, doctor, and mind seeding. Order matters twice: skill apply precedence and composed-persona ordering.

6Materialization (drwn write)

effective state
  ├── generated/workers/<card>/          per-worker bundle
  │     ├── skills/<name>               symlinks into the extracted content root
  │     ├── mcp/servers.json             the card's MCP server definitions
  │     ├── hooks/{claude,codex}/composer.mjs   bundled hook policies (consent-gated)
  │     └── worker.json                  index: name/version/integrity/skills/hooks/servers
  ├── generated/workers.json             global index
  └── downstream projection              .claude/ · .codex/ · .cursor/ (skills links, MCP config)

Write is managed and drift-aware: every emitted path is recorded with a content hash; if a user replaced a managed path with their own content, write refuses (exit non-zero, "drift") unless --force — and even forced, user content is preserved, not clobbered. --dry-run --json reports the full change plan; DRWN_STORE_READONLY=1 blocks all store mutations (honored by every mutating command).

8Deployment

drwn worker deploy <blueprint-ref> --name chief [--secrets-file .drwn.secrets]
  1. CLI resolves the blueprint + members LOCALLY → pinned member set (version/integrity/treeSha)
  2. POST /api/deployments { cardRef, name, secrets?, blueprint: { members, governance } }
  3. ← { deploymentId, mindId, slug }   — mindId = the STABLE worker id (legacy field name;
       minted for every worker, mind or not; survives redeploys/rollbacks)
  4. poll → ready; cache mind binding coordinates (never tokens) in mind-bindings.json

Design decision worth knowing: the CLI resolves, the server materializes. The server receives an immutable pinned member set — no server-side composedFrom resolution, reproducible by construction. Auth is DAH bearer tokens with automatic refresh; MCP secrets upload at deploy and reach containers as MCP_TOKEN_* env. Rollback flips the serving alias between deployments of the same worker; the worker's identity (and mind) persists across all of it.

9The mind subsystem

0.7.0's addition, in one paragraph (doc 113 is the full guide): cards may carry persona/beliefs seeds and memory layer declarations; at provision, the active stack's mind content is composed (stack-ordered, provenance-fenced) and seeded into BeginningDB at minds/<mindId>/, where it becomes the live, DB-first-edited state; memory (L4 reflections, L5 observations) is DB-native via a shared placement pool; sync/diff/checkpoint run the card↔DB loop with DB-edits-win semantics.

Architecturally it lives behind two module seams, deliberately split:

cli/core/mind-content/ — pure, offline

Manifest section validation, persona composition/fence parsing, visibility + push gate, authoring, publish validation. No network. This is the restored (pre-descope) machinery, re-homed behind narrow seams in manifest/lock/publish.

cli/core/mind-store/ — everything network

The BeginningDB client (fetch-only, ETag CAS, offset-write append), path conventions (ULID pool entries), the seed/rebase/checkpoint engines, the seed ledger (mind.json), doctor probes, binding cache. Swappable for the future @beginningdb/client package.

Locks that carry mind content raise the project's version floor to MINDS_MIN_DRWN_VERSION = 0.7.0.

11Versioning & compatibility

DRWN_VERSION (0.7.0) is kept in lockstep with package.json (parity-tested). Locks carry a minDrwnVersion floor: 0.3.0 when any card has hooks, 0.7.0 when any card has mind content — older CLIs warn loudly instead of mis-materializing. Card-level compatibility rides harness.minVersion per manifest. Earlier vocabulary you may meet in old docs: "mind card" once meant harness cards generally (pre-rename, ≤ analysis 98), and bgng was this CLI's former name — everywhere current, bgng is the BeginningDB CLI.
