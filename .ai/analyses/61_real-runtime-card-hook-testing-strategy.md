# Real Runtime Testing Strategy for Drwn Card Hooks

**Date:** 2026-06-12  
**Status:** Draft strategy  
**Scope:** Concrete testing strategy for Drwn Card hooks against real Claude Code, Codex, and Mastra runtimes, including a CCH-backed path.

---

## Executive Summary

The current Card hook implementation is well covered by unit, integration, and CLI E2E tests inside `darwinian-harness`, but it does not yet prove that the generated artifacts are accepted and executed by the live host runtimes. We should add an opt-in live-runtime test layer that verifies the actual host CLIs and Mastra runtime execute Drwn-generated hook composers, while keeping regular CI deterministic and credential-free.

The recommended long-term strategy has two complementary tracks:

1. **Local live-runtime suite in `darwinian-harness`**: opt-in tests that run installed `claude` and `codex` CLIs against a temporary project generated by `drwn`, and run a Mastra fixture locally through CCH's `runtime/run.js`. These tests should be release-gate or maintainer-gate tests, not default `bun test`.
2. **CCH-backed container suite**: an opt-in suite that mounts the same generated hook fixture into CCH Codex and Mastra runtime shapes. This protects container behavior, Linux shell quoting, runtime asset mounting, and `/opt/mastra-runtime` module resolution.

Do not rely on final model text as the assertion surface. Use sentinel files and hook-produced JSONL logs as the source of truth. The model only needs to trigger a small tool call.

### Claude Code Contract

Primary docs:

- Claude Code hooks reference: `https://code.claude.com/docs/en/hooks`
- Claude Code settings: `https://code.claude.com/docs/en/settings`
- Claude Code CLI reference: `https://code.claude.com/docs/en/cli-reference`

Relevant facts:

- Claude Code reads hooks from settings, including project `.claude/settings.json`.
- Command hooks receive JSON on stdin.
- Command hooks can return JSON on stdout with `hookSpecificOutput`.
- `PreToolUse` supports `permissionDecision` values including `allow`, `deny`, `ask`, and `defer`, plus `updatedInput` and `additionalContext`.
- `PostToolUse` runs after successful tool calls and can add context or block subsequent processing, but cannot undo the already completed side effect.
- CLI flag `--include-hook-events` with `--output-format stream-json` exposes hook lifecycle events in the output stream, which is valuable for assertions.
- `--bare` must not be used for these tests because it explicitly skips hooks.
- Command-line `--settings` can inject test-only settings, and `--setting-sources` can limit file-based sources. Managed settings may still apply.

Current implementation alignment:

- Drwn emits `.claude/settings.json` `hooks.PreToolUse` and `hooks.PostToolUse` entries using `type: "command"`, `command: "node"`, `args: [composerPath]`, and `timeout`.
- That is compatible with Claude's documented command-hook shape.

### Codex Contract

Primary docs:

- Codex hooks docs: `https://developers.openai.com/codex/hooks`
- Fresh Codex manual was fetched through the OpenAI docs skill to `/var/folders/3n/d0fmlfyn56x36j439r_fb3gm0000gn/T/openai-docs-cache/codex-manual.md`.

Relevant facts:

- Hooks are enabled by default unless `[features].hooks = false`.
- Codex discovers `hooks.json` and inline `[hooks]` tables next to active config layers.
- Common useful locations include user `~/.codex/hooks.json` and project `.codex/hooks.json`.
- Project-local hooks load only when the project `.codex/` layer is trusted.
- Non-managed command hooks require review and trust by hook hash before they run.
- `codex exec --dangerously-bypass-hook-trust` is intended for automation that already vets hook sources.
- Matching hooks from multiple files all run. Multiple matching command hooks for the same event are launched concurrently.
- `PreToolUse` supports `permissionDecision: "deny"`, `additionalContext`, and `updatedInput` with `permissionDecision: "allow"`.
- Codex explicitly does not support `permissionDecision: "ask"` for `PreToolUse`; returning it is an error and Codex continues the tool call. Drwn correctly degrades portable `ask` to a deny-shaped result for Codex.
- `PostToolUse` currently covers Bash, `apply_patch`, and MCP calls, but not every shell mechanism. Tests should use simple Bash commands.

Current implementation alignment:

- Drwn emits a single project `.codex/hooks.json` with one composite command hook per event.
- The single composite hook avoids Codex's concurrent multi-hook behavior from undermining Drwn's deterministic policy order.
- Project-local Codex tests must handle both project trust and hook trust. Use a temporary `CODEX_HOME` with a copied auth file plus a minimal trusted-project config, then pass `--dangerously-bypass-hook-trust`.

### Mastra Contract

Primary docs:

- Mastra tool hooks guide: `https://mastra.ai/docs/agents/using-tools`
- Mastra Agent reference: `https://mastra.ai/reference/agents/agent`

Relevant facts:

- Mastra `Agent` accepts `hooks` with `beforeToolCall` and `afterToolCall`.
- `beforeToolCall` receives `{ toolName, input, context, metadata }`.
- Returning `{ proceed: false, output }` short-circuits the tool call and uses `output` as the tool result.
- `afterToolCall` receives success output or error; if the tool threw, Mastra rethrows after the hook runs.
- Per-execution hooks can be passed to `.generate()` / `.stream()` and merge with agent-level hooks.

Current implementation alignment and gap:

- Drwn emits `.agents/drwn/generated/hooks/mastra/composer.ts` containing `policies` and re-exporting `composeToolHooks`.
- This is enough for a consumer to wire Mastra hooks manually, but it is not yet a ready-to-assign `hooks` object.
- Long-term stable Mastra integration should add a generated `hooks` export or `createMastraHooks()` helper so consumers and tests do not duplicate adapter glue.

## What Must Be Proven

The live-runtime suite should prove these invariants:

1. `drwn` can create, publish/apply, trust, and write a hook-bearing card into a fresh project.
2. Generated Claude and Codex command composers are executable by the real host runtimes.
3. Generated Mastra composer can be consumed by a real or CCH-compatible Mastra agent bundle.
4. Runtime payloads decode into the expected portable `ToolPolicyEvent` shape.
5. Portable policy decisions encode into effective runtime behavior:
   - `deny` blocks or replaces the tool result.
   - `allow + updatedInput` rewrites simple supported tool inputs where the runtime supports it.
   - `additionalContext` is accepted without corrupting runtime execution.
   - `afterToolCall` observes successful results.
   - observer failures do not derail the run; enforcement failures fail closed.
6. Consent gates work before live runtime execution: untrusted hooks do not materialize; trusted hooks do.
7. Project exclusions suppress a hook before consent checks and before runtime materialization.
8. Managed-content drift protections prevent unreviewed local hook changes from being silently overwritten.

## Shared Fixture Design

Create one reusable fixture generator rather than separate per-runtime fixtures.

Proposed fixture module:

- `test/live/helpers/hook-card-fixture.ts`

Generated card source:

```text
hooks/
  record-tool-call/
    policy.ts
  deny-sentinel/
    policy.ts
  rewrite-sentinel/
    policy.ts
  observer-failure/
    policy.ts
card.json
package.json
```

Policy behavior:

- `record-tool-call`: observer; appends compact JSON to a log file derived from `process.env.DRWN_HOOK_E2E_LOG` or `event.cwd/.agents/drwn/hook-e2e.jsonl`.
- `deny-sentinel`: enforcement; denies Bash/command input containing `DRWN_BLOCK_ME`; synthetic output should include a stable marker.
- `rewrite-sentinel`: enforcement; rewrites a simple Bash command containing `DRWN_REWRITE_ME` into `printf 'DRWN_REWRITTEN\n'`.
- `observer-failure`: observer; intentionally throws only when `DRWN_HOOK_E2E_OBSERVER_FAILURE=1`, proving observer errors are swallowed/logged.

The tests should assert the JSONL log, sentinel files, and stream events. They should not assert exact final model prose.

## Strategy A: Local Live Runtime Suite

### A1. Claude Code Live Test

Suggested file:

- `test/live/claude-card-hooks.e2e.test.ts`

Gate:

- Only run when `RUN_DRWN_REAL_CLAUDE=1`.
- Skip if `claude` is absent.
- Prefer a version floor: `claude --version` must be at least `2.1.153` or the minimum that supports the asserted hook behavior.

Setup:

1. Create a temp git project.
2. Use the fixture generator to create a local card source in an isolated agents dir.
3. Publish/apply/add the card through the `drwn` CLI or direct CLI helpers.
4. Run `drwn card trust <card> --hooks`.
5. Run `drwn write --scope project --target claude`.
6. Confirm `.claude/settings.json` points at the generated composer.

Execution:

Run `claude` with:

```bash
claude -p \
  --output-format stream-json \
  --verbose \
  --include-hook-events \
  --no-session-persistence \
  --setting-sources project \
  --settings '{"permissions":{"allow":["Bash(printf *)","Bash(touch *)"]}}' \
  --tools Bash \
  --allowedTools 'Bash(printf *)' 'Bash(touch *)' \
  --permission-mode acceptEdits \
  --max-turns 3 \
  'Use Bash to run exactly: printf "DRWN_REWRITE_ME\n"'
```

Use a second deny case with a command that would create a sentinel file if not blocked:

```bash
touch DRWN_BLOCK_ME_created_by_tool
```

Assertions:

- `hook-e2e.jsonl` contains `runtime: "claude-code"`, `phase: "pre-tool"`, and `toolName: "Bash"`.
- Rewrite case records an updated input and the runtime sees the rewritten output marker.
- Deny case does not create the sentinel file.
- Stream JSON contains hook lifecycle events when `--include-hook-events` is active.

Stability notes:

- Do not use `--bare`; it skips hooks.
- Use `--no-session-persistence` to reduce local state.
- Use `--setting-sources project` to reduce user settings interference. Managed settings may still apply; tests should surface that as an environment issue.
- Keep prompts small and imperative, but assert only hook logs and files.

### A2. Codex CLI Live Test

Suggested file:

- `test/live/codex-card-hooks.e2e.test.ts`

Gate:

- Only run when `RUN_DRWN_REAL_CODEX=1`.
- Skip if `codex` is absent.
- Require a usable auth source:
  - either existing `~/.codex/auth.json`, or
  - `CODEX_AUTH_JSON_B64` for CI.
- Model configurable through `DRWN_LIVE_CODEX_MODEL`, defaulting to the same small model used by CCH once verified.

Setup:

1. Create a temp project and initialize git.
2. Create a temp `CODEX_HOME`.
3. Copy or decode auth into `$CODEX_HOME/.codex/auth.json`.
4. Write `$CODEX_HOME/.codex/config.toml` with:

```toml
[features]
hooks = true

[projects."/absolute/temp/project"]
trust_level = "trusted"
```

5. Generate/apply/trust/write the fixture card with `--target codex`.
6. Confirm project `.codex/hooks.json` points at the generated composer.

Execution:

Run from the temp project:

```bash
CODEX_HOME="$TEMP_CODEX_HOME" codex exec \
  --json \
  --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  --dangerously-bypass-hook-trust \
  -C "$PROJECT" \
  -m "$DRWN_LIVE_CODEX_MODEL" \
  -c 'model_reasoning_effort="low"' \
  'Use Bash to run exactly: printf "DRWN_REWRITE_ME\n"'
```

Assertions:

- The hook log contains `runtime: "codex"` and Bash `PreToolUse`.
- Rewrite case logs the updated input and does not emit Codex hook errors.
- Deny case does not create the sentinel file.
- `codex exec --json` stream includes normal tool events and no hook-trust warning when bypass is used.

Stability notes:

- Do not use the real user `CODEX_HOME` directly; user global hooks can interfere.
- Do not pass `--ignore-user-config` in this setup, because it would also ignore the temporary trusted-project config.
- Use `--dangerously-bypass-hook-trust` only inside this vetted test fixture.
- Keep only one Drwn-generated command hook per event; Drwn's composite hook preserves deterministic ordering, while Codex itself starts multiple matching hooks concurrently.

### A3. Local Mastra / CCH Wrapper Test

Suggested file:

- `test/live/mastra-card-hooks.e2e.test.ts`

Gate:

- The deterministic wrapper test can run without provider credentials once wired.
- A true provider-backed smoke should be gated by `RUN_DRWN_REAL_MASTRA=1` and provider credentials such as `OPENROUTER_API_KEY`.

Setup:

1. Generate/apply/trust/write the fixture card with Mastra runtime enabled:

```json
{
  "hooks": {
    "runtimes": {
      "mastra": true
    }
  }
}
```

2. Build a temporary agent module under a CCH-style runtime root:

```text
$runtimeRoot/
  agents/drwn-hook-fixture/index.js
  node_modules -> /Users/pureicis/dev/containerized-cli-harness/packages/mastra/node_modules
```

3. The agent module imports the generated composer and wires it to a compatible `.generate()` method.

Deterministic agent shape:

```js
import { policies, composeToolHooks } from "/abs/project/.agents/drwn/generated/hooks/mastra/composer.ts";

const composed = composeToolHooks(policies, { runtime: "mastra", logger: console });

export default {
  async generate(prompt, options) {
    const pre = await composed.beforeToolCall({
      runtime: "mastra",
      phase: "pre-tool",
      toolName: "bashTool",
      input: { command: prompt },
    });
    if (pre?.action === "deny") {
      return {
        object: options.structuredOutput.schema.parse({ ok: true, blocked: true }),
        finishReason: "stop",
      };
    }
    await composed.afterToolCall({
      runtime: "mastra",
      phase: "post-tool",
      toolName: "bashTool",
      input: { command: prompt },
      output: { stdout: "ok" },
    });
    return {
      object: options.structuredOutput.schema.parse({ ok: true, blocked: false }),
      finishReason: "stop",
    };
  },
};
```

Execution:

Run CCH's local wrapper:

```bash
MASTRA_RUNTIME_ROOT="$runtimeRoot" \
node /Users/pureicis/dev/containerized-cli-harness/packages/mastra/runtime/run.js \
  --agent-id drwn-hook-fixture \
  --schema-path "$schema" \
  --output-path "$output" \
  --events-path "$events"
```

Assertions:

- Wrapper exits 0.
- Output JSON matches schema.
- Event JSONL ends in `agent.completed`.
- Hook log records Mastra pre/post events.
- Deny case returns the blocked object without executing the fake tool path.

Recommended follow-up before calling this fully mature:

- Enhance `emitMastraComposer` to export a ready Mastra-compatible `hooks` object, or at least `createMastraHooks(logger?)`. That removes duplicate adapter glue from every consumer and from tests.

## Strategy B: CCH-Backed Container Suite

This is the best long-term confidence layer for container behavior, but it should stay opt-in because it needs Docker/Cloudflare-style runtime setup and provider credentials for Codex.

Suggested package/location options:

- In `darwinian-harness`: `test/cch/*.test.ts` that imports CCH source packages by sibling path.
- Or in CCH: a new fixture consumer package that depends on `darwinian-harness` and exercises hook asset mounting.

### B1. CCH Codex Hook Test

Required CCH test harness shape:

- Subclass `CodexCliRuntime` for the test.
- Override `mountPreExecAssets` to:
  - copy generated `.agents/drwn/generated/hooks/codex/composer.mjs` into `/workspace/.agents/drwn/generated/hooks/codex/composer.mjs`,
  - write `/workspace/.codex/hooks.json`,
  - write or append `/opt/codex-home/.codex/config.toml` trusted project entry for `/workspace`,
  - create a writable hook log path.
- Override or extend `buildCliCommand` to include `--dangerously-bypass-hook-trust`.

Long-term CCH API improvement:

- Add an opt-in runtime config flag such as `runtimeConfig.codex_bypass_hook_trust === true` for vetted automation.
- Optionally add a generic `runtimeConfig.pre_exec_files` asset map if several consumers need file mounting.

Assertions:

- Same as local Codex test, but validate the paths inside `/workspace`.
- Verify no user hooks are loaded by using the CCH-owned `CODEX_HOME`.
- Use one container/DO per run, preserving CCH's existing isolation finding.

### B2. CCH Mastra Hook Test

Required CCH test harness shape:

- Use CCH's Mastra runtime root contract:

```text
/opt/mastra-runtime/
  run.js
  node_modules/
  agents/drwn-hook-fixture/index.js
```

- Build an agent bundle that imports the generated Mastra composer.
- For deterministic tests, a generate-able object is enough.
- For provider smoke, use a real `@mastra/core/agent` Agent with one simple tool and Drwn-provided hooks.

Assertions:

- The wrapper imports the agent.
- Hook policies execute before and after a tool call.
- Block result maps through `{ proceed: false, output }` in the real Agent smoke.
- CCH trajectory contains step and completion events.

## Recommended Implementation Order

1. **Add contract canary fixtures** for Claude/Codex/Mastra payload and output examples. These stay default-CI safe.
2. **Add the shared hook-card fixture generator** and use it in direct composer tests first.
3. **Add local Claude live test** using `--include-hook-events` and sentinel files.
4. **Add local Codex live test** using temp `CODEX_HOME`, trusted temp project, and `--dangerously-bypass-hook-trust`.
5. **Improve Mastra composer output** with `createMastraHooks()` or `hooks` export.
6. **Add local Mastra wrapper test** using CCH's `runtime/run.js` and a deterministic generate-able object.
7. **Add provider-backed real Mastra Agent smoke** as opt-in.
8. **Add CCH Codex/Mastra container tests** once local live tests are stable.

## Known Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Model may not call the intended tool | Use imperative prompts, low max turns, simple Bash-only tool availability, and assert hook logs rather than final prose. |
| User/global hooks interfere | Use temp project settings for Claude; use temp `CODEX_HOME` for Codex; CCH uses isolated home paths. |
| Codex project trust and hook trust are different gates | Explicitly trust the temp project in temp config and pass `--dangerously-bypass-hook-trust` for automation. |
| Live CLI tests consume model credits | Keep them opt-in and release-gate only; use smallest verified model aliases. |
| Claude managed settings can still alter behavior | Detect and report as an environment precondition failure, not a product failure. |
| Codex `PostToolUse` coverage is incomplete for richer shell mechanisms | Use simple Bash commands only. |
| Mastra composer currently requires manual wiring | Add generated `hooks` or `createMastraHooks()` export before broad adoption. |
| CCH Codex currently omits hook-trust bypass | Use test subclass first; add opt-in runtime config only if more consumers need it. |
| Container tests can be slow/flaky | Keep opt-in, one sandbox per run, bounded timeouts, and deterministic non-provider Mastra test as the baseline. |

## Acceptance Criteria for "Real Runtime Tested"

Call Card hooks real-runtime tested only when all are true:

- Existing `bun run typecheck` and `bun test` pass.
- Contract canaries pass in default CI.
- `RUN_DRWN_REAL_CLAUDE=1` live Claude test passes on a clean temp project.
- `RUN_DRWN_REAL_CODEX=1` live Codex test passes with temp `CODEX_HOME`.
- Local CCH Mastra wrapper test passes without provider credentials.
- A real `@mastra/core` Agent smoke passes behind `RUN_DRWN_REAL_MASTRA=1`.
- CCH Codex/Mastra container tests have either passed in a release environment or are explicitly marked as not yet part of the release gate.

---

## Bottom Line

The stable path is not to make every normal unit test launch AI runtimes. Keep default CI fast and deterministic, then add a small opt-in live suite that verifies actual host acceptance and behavior using sentinel artifacts. For CCH, reuse `mountPreExecAssets` and `/opt/mastra-runtime` rather than adding Drwn-specific coupling to CCH. The one implementation improvement that would materially reduce integration risk is to make the generated Mastra composer export a ready-to-use Mastra `hooks` object.
