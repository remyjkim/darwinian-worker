Org↔Worker Instantiation Interface

The Architect's accepted design artifacts — Org Blueprint and MemoryPlan — stay exactly what they are; a sibling WorkerSpec section carries the one convergent addition (runtime + capability grants with provenance); and a single governed contract, org-worker-bundle-v1, compiles the accepted design into drwn inputs. drwn remains the sole materializer, its receipts feed the org readiness gate, and two projection systems become one boundary. This deep dive carries the full mapping: every instantiation category, both diagrams, the five contract shapes, and one worker traced end to end.

01

Drivers

Two teams, days from building the same seam twice, incompatibly — while the gaps each side names are mechanized on the other side already.

Architecture driver

Two instruction projectors are about to exist, and worker capability authority is recorded nowhere

Believer §03 plans planToWorkerBundle to emit per-worker CLAUDE.md, a capabilities manifest, and a card stack by hand-shaped compilation. drwn's approved architecture (analysis 124, decided 072226) projects worker instructions to a canonical root AGENTS.md block — grounded in cross-agent research showing root CLAUDE.md double-reads in Cursor — with per-card consent through the card-trust ledger. Built independently, these drift by construction and violate the single-composer criterion both teams adopted on their own.

The mirror-image gaps

Each side's register names problems the other side has already shipped answers to

The org gap register: tool authority is "granted by hand and recorded nowhere" (G3); "is this org ready to start?" has no mechanical answer (G6); runtime binding is hand-authored per org (G2). The drwn side ships the mechanisms these describe — a semver-ranged consent ledger, idempotent receipted writes, and a locked capability record — but at worker scale, with nothing org-side consuming them. The interface exists so neither side rebuilds the other's half.

03

Target architecture

Artifacts stay put, one schema addition gains a concrete shape, and one contract carries everything across the seam.

1Artifacts

Architect outputs unchanged; the planned additions land as a sibling WorkerSpec

The Org Blueprint (org shape) and MemoryPlan (memory topology) remain the accepted design artifacts exactly as the Architect program defines them — nothing below this boundary reaches back into them. The one convergent change refines where believer §03 planned it: rather than widening MindSpec (which is purely a memory-topology spec — views, seeds, knowledge strategy), a sibling WorkerSpec section carries the worker-plane concerns — runtime and capabilities with grant provenance, {card, capability: hooks | instructions, semverRange, grantedBy} — keyed by the same entity id, with the org's doctrine ratifier as grantedBy. Capabilities belong to the worker, not to its memory; this keeps the Memory Blueprint honest, sidesteps the worst of the mind homonym (register C6), and makes the amendment target literally worker.capabilities. It mechanizes G3 and satisfies drwn's decided consent gate in one move; because tool authority is authority, the section sits under the existing amendment gate classes, exactly as the believer proposal intends. (Fallback if the org side prefers one-spec-per-member: rename the widened type per C6 — decided at the session.)

2Contract

org-worker-bundle-v1 — five compiled inputs, in drwn's vocabulary

Following the Architect program's contract idiom (producer, consumers, truth carried): producer — the bundle compiler; consumers — the drwn CLI and, through its receipts, the readiness gate; truth carried — which cards, which roots, which settings, whose authority, which version floor. The bundle emits references and selections, never content bodies and never harness files. Workspace scaffolds, env/secret material (op:// references), operator wiring, and runbooks stay org-side — drwn only ever references environment variables through its per-target interpolation and never stores secret material in cards, locks, or projections.

3Execution & receipts

Foundry runs the drwn sequence; receipts flow back to the readiness gate

Inside the scaffolded workspace: drwn init --non-interactive → apply the five inputs → drwn install --frozen → drwn write --dry-run (preview receipt) → drwn write --strict. The write-record and JSON output are captured as the worker-materialization receipt feeding the org readiness gate (G6); a failed strict write halts the gate rather than recording-and-continuing. drwn doctor is the standing conformance check after Day-0, and repeated writes are byte/mtime-idempotent, so receipts are stable across re-runs — matching the bundle pipeline's "idempotent, receipted stages" doctrine.

07

Assumptions and open questions

The session agenda, with materiality. None block the manual pilot; all block the contract freeze.

01

Org workers materialize at project scope — one workspace per worker

The five-input whitelist (input 3) assumes project-scope writes. A machine-scope org worker would change the overlay shape and the receipt path.

Material
02

Ratification state can auto-satisfy trust for org-catalog cards

drwn's trustedSources policy could treat the org catalog as a trusted source, making grants declarative rather than per-command. If not, Foundry issues explicit trust commands per grant (as in §04c) — workable, just noisier.

Bounded
03

Network fetch is available at instantiation

Air-gapped instantiation would require the bundle to carry card archives (drwn's portable inventory transfer exists for this). Assumed unnecessary for the pilot.

Bounded
04

The AGENTS.md user section starts empty in org workspaces

Whether an org seed (charter-derived preamble) should populate the user-owned section is an org-side authoring question; the drwn block owns only its markers either way (§04e).

Bounded
05

A failed strict write halts the readiness gate

Record-and-continue would let a worker pass readiness with a known-drifted or consent-failing materialization. Halt is assumed; the org side owns the gate policy.

Material
06

The pilot can precede drwn's phase-1 instructions projection

Until phase 1 lands, worker instructions reach sessions through the instructions-skill floor (weaker, pull-strength). If the pilot requires push-strength instructions on day one, phase-1 sequencing becomes coupled — currently assumed decoupled.
