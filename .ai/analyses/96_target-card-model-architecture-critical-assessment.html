Analysis 96 — Critical Assessment of the Target Card Model Architecture (Analysis 93)

Date: 2026-07-03  ·  Author: Claude + Remy  ·  Status: Assessment — proposes amendments to analysis 93 and tasks/65 before Stage B execution

1. Method and stance

Analysis 94 already red-teamed the model. This assessment is second-order: it audits the consolidated document — including the folded-in 94 corrections — against three grounds it can be wrong on:

  1. The code as it exists. Every present-tense claim in 93 was checked against the implementation (including the uncommitted Stage-A changes), with file:line evidence.
  2. The source analyses it consolidates. What did 82/89/90/92/94 decide that 93 dropped, weakened, or mutated in transit?
  3. Itself. Do the four principles, the figures, the flows, and the rollout table agree with each other?

Findings are graded: CONFIRMED (claim verified sound), DEFECT (claim contradicted by code or by the doc itself), GAP (load-bearing mechanism under-specified), DROPPED (consolidation loss vs source analyses), RISK (strategic exposure). Severity is HIGH / MED / LOW by impact on Stage-B correctness.

2. Verdict up front

The skeleton survives; the document does not fully earn its own claims. Immutable content + mutable metadata + provenance-as-data + per-project activation is verified sound where it is already built, and the rejected alternatives stay rightly rejected. But the audit found: two HIGH defects (an integrity guarantee stated in the present tense that the code does not provide; a silent behavioral inversion of skill-conflict precedence that no document acknowledges), one HIGH design contradiction (the profile card — the flagship replacement for machine defaults — has no team-compatible activation channel), and one HIGH mechanism gap (the union-merged meta ref, the doc's most novel contribution, has no defined semantics for retraction, scalar conflict, or concurrent push). None of these invalidates the architecture; all four must be resolved in 93/65 before their respective Stage-B phases, because each is cheapest to fix on paper.

4. Defects — the doc vs the code and vs itself

D1. The integrity chain records; it does not verify DEFECT HIGH

93 §4.3 annotates the lockfile's integrity field "verified on resolve", and the §5.3 consume flow says "card clone <url> … integrity verified." Neither is true, today or as designed for Stage B:

  • computeCardIntegrity is called at publish (card-store.ts:800), at resolve (card-store.ts:692), and at file-origin resolve (card-store.ts:849) — and the result is recorded into the commit message and the lock entry (card-project.ts:46). Nowhere in the codebase is a freshly computed hash compared against a previously trusted value. There is no verify-on-load, no re-check after extraction, no clone-time verification.
  • Even conceptually, there is nothing to verify against until catalog v2 supplies per-version integrity from an independent channel — and catalog v2 is Stage C (LATER). Until then the chain is trust-on-first-use with no subsequent re-use of the trust anchor: the weakest form, presented in the strongest language.
Why HIGH: this is a security property claimed in the present tense in the architecture of record. Anyone reading 93 — including a future agent operator — will assume tamper-evidence that does not exist. Either (a) the wording changes to "recorded at publish/resolve; verification lands with catalog v2", or (b) a cheap verify-on-resolve (lock integrity vs recomputed extraction hash) is added to Stage B. Option (b) costs one comparison in resolveCard and would make the sentence true; recommend both.

D2. Skill-conflict precedence: the target silently inverts current behavior DEFECT HIGH

93 §5.6 (and 94 §4.5, and 65 Phase 4) decree later-apply wins, "matching mind-stack layering." The code today does the opposite: the skill resolver walks cards in lockfile order and first wins — asserted by an existing test whose title says exactly that (test/core-card-skill-resolver.test.ts:67 — "resolveSkillSource walks cards in lockfile order on conflict (first wins)").

No document in the chain (93, 94, 65) acknowledges that the "new deterministic rule" is a reversal of an existing deterministic rule. Consequences:

  • When 65 Phase 4 lands, any existing project with overlapping cards silently flips which card's copy materializes on the next drwn write. That is precisely the class of surprise the drift gates exist to prevent.
  • 65 Phase 4 says "implement deterministic precedence" as if none existed, and its test plan contains no migration/behavior-change case. The existing first-wins test will simply fail and be rewritten — which is how behavior changes sneak past review.
Amendment: 93 §5.6 and 65 Phase 4 must state the inversion explicitly; Phase 4 needs (a) a changelog-visible note, (b) a one-time warning when a write's conflict winner differs from what the previous write-record materialized, and (c) a decision from Remy that later-wins is in fact wanted over first-wins (the mind-stack analogy argues for it, but it should be a decision, not an accident of consolidation).

D3. Figure 2b draws the symlink the doc rejects DEFECT MED

93's rejected-alternatives list rules out symlink materialization (Windows/Cowork, hash drift gates), and 65's cross-cutting gate says "No new symlinks in any write path." Yet Fig 2b renders the generated minds layer as skills/* → extracted/<sha>/skills/* — a link — and the code still carries symlink/generated-symlink managed-path kinds (sync.ts:152, 266; the "pre-de-symlink" cleanup comment at sync.ts:369 confirms the migration is mid-flight). Analysis 94 §3.6 honestly lists this as residual debt; 93 draws the debt as the target with no label. A target-architecture figure that depicts a state its own invariants forbid will mislead every future implementer. Label the arrow as pending-debt (82 fast-follow) or draw the copy.

D4. Small internal inconsistencies DEFECT LOW

  • write-record.json location. Fig 1 places it inside generated/ ("generated/ minds layer · write-record.json"); Fig 2b and the code put it at .agents/drwn/write-record.json (write-record.ts:63–65). Fig 1 is wrong.
  • "init-managed into .gitignore" (93 §5.1) is stated as an existing mechanism; today init only warns about .gitignore, never edits it (commands/init.ts:77–85). The behavior is planned (65 Phase 3 Step 1) — the doc should mark it as such.
  • Porcelain verb naming drift. 93 §5.6 shows drwn release <card>; 65 Phase 2 implements drwn card release while its own success criteria say drwn release. Pick one before the operator skills teach it.
  • "(schema v2, decided Q3)" is ambiguous. It means decision-question 3, but reads as calendar Q3 — the consolidation reviewer in this very assessment misparsed it. Write "decided question 3."

5. Gaps — load-bearing mechanisms that are named, not designed

G1. "Union-merge" is a slogan, not a merge semantics GAP HIGH

The meta ref is 93's most novel mechanism and its correctness rests entirely on the phrase "union-merged." Three cases the phrase does not cover:

  • Retraction is unrepresentable. A pure union can only grow. Un-deprecating a version (a mistaken card deprecate — inevitable) cannot propagate: every other machine's copy re-contributes the entry at the next merge and the deprecation resurrects forever. The schema needs tombstones or per-key last-write-wins with an ordering field — and the at timestamp already in the deprecation entry is unused by any stated merge rule. 65 Phase 5 says "last-write-wins only within a single key" without saying ordered by what.
  • successor is a scalar. Two machines setting different successors is not a union-mergeable situation; the doc defines no resolution. (It is also the trust-sensitive field — see G6.)
  • fetch → merge → push is not atomic. Two writers can interleave between fetch and push and the second push either fails (then what — retry loop? how many times?) or, if implemented as a plain update-ref on the remote, silently discards the first writer. This needs compare-and-swap semantics (--force-with-lease-style old-value assertion) with a bounded retry. 65's test plan covers only sequential deprecations from one machine — the exact case that never exercises the merge.
Amendment: specify metadata merge as: per-key LWW ordered by the at timestamp (with machine-id tiebreak), deletions as tombstoned keys, successor as an {value, at} LWW register, and push as compare-and-swap with retry. This is ~15 lines of spec that turns "never blind force" from an aspiration into an algorithm — and it must precede 65 Phase 5, whose tests should include the concurrent-writer case.

G2. The profile card has no team-compatible activation channel GAP HIGH

Principle 1 retires machine defaults into "a profile card applied per project." But per-project activation writes to config.json — which 93 itself defines as shareable intent, committed — and pins into card.lock. Apply @remyjkim/everyday in any team repository and:

  • every teammate's install-project tries to resolve a personal card they may have no access to (or simply do not want), and
  • the personal staples become project intent, reviewed in PRs, forever.

config.local.json does not solve this: 93/65 scope it strictly to link overrides of already-applied cards, not local-only activation. So the flagship migration of Stage B (Phase 6) — the replacement for the retired defaults channel — currently composes only with solo repositories, and neither 93 §7 nor 65 Phase 6 mentions the constraint.

Amendment: either extend config.local.json with a local cards list (locally-activated, never locked into the shared card.lock — which raises its own question of where their pins live), or explicitly scope the profile card to solo/personal repos in 93 §7 and the migration runbook. This is a design decision for Remy, not an implementation detail — it changes what "retire machine defaults" means for team projects, where the honest answer today may be "defaults had no team story either, and neither does this."

G3. Release pipeline: failure states enumerable, recovery unspecified GAP MED

93 §5.2 says only "stops at first failure." 94 §4.7 explicitly required "every step idempotent, re-run continues," and 65 Phase 2 carries that requirement — so 93, the architecture of record, is the odd one out. The partial states are worth enumerating because each strands a different component: bump committed but publish failed (source ahead of store); published but push failed (store ahead of remote); pushed but catalog failed (remote invisible to discovery). One more wrinkle nobody records: the version-bump commit lives in the card source's own .git, which is machine-local and unsynced (94 §3.4) — publish history is losable in a way the bare card repo is not. Fold 94 §4.7's resumability sentence back into 93 §5.2 and note the source-repo provenance caveat under the multi-machine open front.

G4. exclude cannot express the resolution it exists for GAP MED

The duplicate-skill escape hatch is "an explicit exclude list in project config." But skills.exclude is name-scoped (types.ts — a flat string list): excluding apply-mind-card drops the name from every card. The rule's whole purpose is "take this skill from card X, not card Y," which requires a card-qualified form (@scope/name#skill). As specified, the escape hatch can only turn the skill off entirely — 65 Phase 4's test ("honors a project skills.exclude entry to drop one deterministically") would pass while the feature fails its purpose.

G5. Upstream refs: the optional revision reintroduces the ambiguity it fixed GAP MED

  • 94 §4.1 added @rev because rev-less refs made sync --check ambiguous ("stale against what?"). 93 §4.1 then makes rev optional with "absent = track default branch" — restoring the moving-target semantics for every unpinned entry. Fine as a deliberate two-mode design, but the doc should say which mode is the norm and that --check against an unpinned ref is inherently racy.
  • 65 validates upstream keys ⊆ include, but not the converse — a bundled skill with no upstream entry is a silent provenance hole, the exact thing Principle 3 exists to abolish. doctor should at minimum warn on include-without-upstream.
  • The sync clone/fetch cache ("resolve the git ref into a cache", 65 Phase 1) appears nowhere in 93's storage layout (Fig 2a). Small, but the layout claims to be the registry of record.

G6. Successor trust-scoping is weaker than its callout sounds GAP MED

The mitigation for the successor attack is "auto-suggest only same-scope successors." But a scope is a string prefix with no ownership binding anywhere in the system — no signing, no namespace registry, nothing ties @darwinian/* cards to one key or one owner. The threat model in the callout is "a compromised remote" — and a compromised remote for one @darwinian card is, from the consumer's perspective, exactly as authoritative about the scope as any other. Same-scope gating adds friction against cross-scope phishing, which is real but the easy half; it does nothing against the stated threat. The honest fix (signing) is deferred to Phase 8 as a document. Keep the gate, but the callout should say what it actually buys, and the trust roadmap should name successor pointers as a signing-dependent feature, not a solved one.

G7. Update policy vs deprecation is undefined GAP LOW

card outdated "warns on deprecated pins" (93 §5.3) — but does card update (and therefore drwn up, which runs unattended across the whole project) re-pin onto a version that is the latest-in-range and deprecated? npm's answer (warn, install anyway) is defensible; skipping to the newest non-deprecated in-range version is also defensible. Pick one and write it down before drwn up automates it.

G8. The store grows forever; nix is cited without nix's GC GAP LOW

93 invokes nix's read-only-store reasoning for immutability, but store gc today only runs git gc per bare repo (commands/store/gc.ts:23–25) — nothing ever prunes extracted/, and 94 §3.6 admits it grows unboundedly. Content-addressed stores are only viable long-term with reachability-based GC (roots = every project's card.lock paths + sources). Not urgent; but a target architecture that names its storage "the immutable ledger" should at least reserve the GC design, because lock entries point into extracted/ (93 §4.3 path) and naive pruning would break resolution.

G9. refs/meta reaches only drwn-shaped consumers GAP LOW

93 §3.2 correctly notes drwn controls refspecs "on both ends" — which is exactly the limit: a plain git clone of a card repo (the doc's own distribution story is "any git host") fetches no refs/meta/*, some hosts restrict non-standard ref namespaces, and shallow clones complicate it further. Deprecation metadata is therefore invisible to every consumer who is not the drwn CLI. Acceptable — but it means the catalog (Stage C) is the only channel that can reach everyone, so catalog-reflected deprecation should not be thought of as a nice-to-have duplicate; it is the fallback of record.

7. Strategic assessment RISK

  • The moat claim is self-undermining as written. "The meta ref is the reserved home for fitness/eval annotations… No competitor is positioned to copy this" (93 §8). Set aside the unfalsifiable competitor claim; the concrete problem is that the doc's own trust analysis (successor pointers as attack surface, unsigned refs, instruction-injection via skills) applies with more force to fitness data than to deprecations. A fitness signal in an unsigned, mutable, union-merged ref is a rankings system anyone who can push can game — the moment catalogs open, exactly when the moat is supposed to matter. Fitness annotations need the signing story more than deprecation does; the Stage-B schema reservation is fine, but the roadmap should sequence fitness-in-meta after signing, and say so.
  • Porcelain is asserted, not measured. 94 §3.1 computed that net concept count is "roughly flat" after the target model; 93 §5.6 presents three or four verbs as the answer to the "single biggest adoption risk" with no success criterion. The claim is plausible (git's precedent is real) but the doc should state the test: e.g., bootstrap-project → drwn use → value in under ten minutes with ≤4 nouns encountered — 94 §5 already names this bar; 93 dropped it.
  • Stage B is eight workstreams wearing one badge. The rollout table's single "NEXT" row spans provenance, porcelain, links, conflicts, distributed metadata, defaults retirement, and hardening. Tasks/65 mitigates properly (independently shippable phases, dependency-honest reordering) — but 93's table hides that the ordering is load-bearing (conflict rule before profile card; merge semantics before distributed deprecation). A reader of 93 alone cannot see which arrows are blocking.
  • The agent-as-operator lens, 94's best insight, is thin in 93. 94 §1 established that the primary operator is an LLM agent and every error must name the next correct command; 93 carries this only inside §5.5's signpost flow. It deserves to be a named principle — it is arguably the strongest justification for provenance-as-data in the whole design.

9. Closing judgment

Analysis 93 is a good consolidation of a sound architecture, and analysis 94's red-team made it substantially better before this assessment touched it. What 94 could not do — being written by the model's own author against the model — is audit the document as an artifact: whether its present-tense claims are true in the code, whether its figures obey its invariants, and whether the consolidation preserved its sources. On that audit, 93 currently over-claims one security property, hides one behavior reversal, ships one team-incompatible flagship migration, and leaves its most novel mechanism specified by a two-word slogan. All four are paper fixes this week and expensive incidents next quarter. Amend first; then Stage B can execute on a document that is as trustworthy as the store it describes.
