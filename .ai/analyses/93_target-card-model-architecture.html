1. The model at a glance

The target card model rests on four load-bearing principles. Everything else in this document is a consequence of them.

  1. Global registry, per-project activation. The machine store under ~/.agents/drwn/ is the single registry of everything available (sources, published versions, catalogs). Whether a card is applied, written, and used is decided per project via card apply + drwn write. Machine-wide activation (the old defaults channel) is retired; personal staples live in a profile card applied per project.
  2. Immutable content, mutable metadata. A published version never changes (git commit + tag, content-addressed extraction). Facts about versions that change over time — deprecation, successor pointers — live in a dedicated mutable ref (refs/meta/*) that travels with push/clone.
  3. Provenance is data, not convention. Every bundled skill carries an upstream ref naming its canonical home. Any copy, at any layer, can answer “where do I edit this?” by walking recorded pointers.
  4. Copies at the edges, ceremony-free loops. Materialization stays copy-based (OS-uniform primitives, per analysis 82). The cost of copying is paid back by two orchestrators: card link (dev loop) and card release (publish loop).

4. The data model

4.1 Card manifest with upstream provenance (recommendation 1)

// sources/@darwinian/operator/card.json  (target shape)
{
  "name": "@darwinian/operator",
  "version": "1.0.1",
  "description": "Primary Darwinian Minds card …",
  "skills": {
    "include": ["apply-mind-card", "inspect-minds", "…"],
    "upstream": {
      "apply-mind-card":
        "git+https://github.com/remyjkim/darwinian-minds-skills.git#skills/apply-mind-card@v0.4.0",
      "inspect-minds":
        "git+https://github.com/remyjkim/darwinian-minds-skills.git#skills/inspect-minds"
      // form: git+URL#subpath[@rev]  — rev optional (tag/commit); absent = track default branch
    }
  },
  "servers": {},
  "harness": { "minVersion": "0.7.0" }
}
  • Canonical ref form is a git URL + subpath, plus an optional revision (@<tag|commit>) — the frozen-snapshot relationship is cross-repo in practice (knowledge-docs ← darwinian-minds). Absent revision tracks the default branch; a pinned revision makes sync --check unambiguous (analysis 94 §4.1).
  • Local paths are dev-only; card publish rejects or rewrites them (consistent with the check-no-local-paths CI rule).
  • drwn card source sync [--check] reads this map and replaces every repo’s private sync script; doctor fails stale copies at publish time, and surfaces “upstream moved” (repo rename / redirect) as a warning, not a hard fail.

4.2 Mutable metadata document (Stage B)

// refs/meta/cards → metadata.json
{
  "metadataVersion": 1,
  "card": "@darwinian/harness-skills",
  "successor": "@darwinian/operator",          // card-level rename pointer (trust-scoped)
  "deprecations": {
    "0.2.0": { "message": "Renamed to @darwinian/operator", "at": "2026-07-02T…" }
  }
}

Consumers see this on every fetch/clone; card show, card list, card outdated, card apply, and card meta show surface it (the ref is invisible in any web UI, so the CLI must show it). Stage A (shipped) stores the same facts in local git config via deprecationKey(); Stage B migrates markers on first write through the same helper, so no debt survives.

Successor pointers are an attack surface (analysis 94 §4.3): a compromised remote could set successor: @evil/operator and make the tooling itself recommend the attacker’s card. Auto-suggest only same-scope successors; cross-scope requires catalog corroboration or explicit user confirmation.

4.3 Lockfile entry (unchanged role, richer chain)

// .agents/drwn/card.lock (excerpt)
{
  "name": "@darwinian/operator",
  "requested": "@darwinian/operator@^1.0.0",
  "version": "1.0.1",
  "integrity": "sha256-…",                      // verified on resolve
  "path": "~/.agents/drwn/extracted/<sha>",
  "origin": "store",
  "git": { "commit": "…" }
}

4.4 Catalog entry, schema v2 (decided Q3)

{
  "name": "operator",
  "url": "git+https://github.com/remyjkim/darwinian-operator.git",
  "description": "…",
  "channels": { "latest": "1.0.1", "stable": "1.0.1" },
  "versions": {
    "1.0.1": { "integrity": "sha256-…" },
    "1.0.0": { "integrity": "sha256-…", "deprecated": "superseded by 1.0.1" }
  }
}

One entry per card; versions and stability channels inside it; integrity per version so search → clone can verify what it fetched (recommendation 5). Until schema v2 lands, releases auto-replace the single pointer entry; git tags keep old versions installable.

5. How the operations work

5.1 Dev loop — card link MODEL RELEASE

Per-card override with a bulk convenience form (decided Q1; location corrected by analysis 94 §4.2). The override lives in a machine-local config.local.json (init-managed into .gitignore) — never config.json (the shareable intent file) and never card.lock. So a teammate cloning the project resolves the pinned version and never sees your absolute local path.

editcanonical skill or card source in your working tree
→
card link @scope/name file:<dir>records override in config.local.json (gitignored; never the lock)
→
drwn write [--watch]re-copies from the live tree; surfaces stay copy-based
→
drwn statusflags card loudly: dev-linked (override)
→
card unlinkrestores the published pin from card.lock

5.6 Porcelain over plumbing MODEL RELEASE

Everything above is plumbing. The single biggest adoption risk (analysis 94 §3.1, §4.6) is concept count — a user must learn ~15 nouns before value. Git survived the same problem by splitting porcelain from plumbing; drwn adds three or four porcelain verbs over the unchanged data model, and the operator skills teach these first:

drwn use  <card-ref>   # clone-if-needed + apply + write
                       #   → one command to value
drwn up                # outdated + update + write,
                       #   across the whole project
drwn dev  <card> <dir>  # link + write --watch
drwn dev --off         #   → unlink, restore pin
drwn release <card>     # the §5.2 publish pipeline,
                       #   bump proposed from card diff

Duplicate-skill conflict rule (analysis 94 §4.5) — a guardrail that must ship before the profile card, since profile + operator cards will bundle overlapping skill names:

  • Deterministic precedence by apply order (later wins), matching mind-stack layering.
  • A loud per-skill warning at write naming both cards.
  • An explicit exclude list in project config for intentional resolution.

Decided questions (2026-07-02)

QDecision
1. card link granularityPer-card primitive; bulk convenience (--all-from) for monorepos. Override location corrected by 94 §4.2: machine-local config.local.json, not config.json.
2. Machine defaultsGlobal registry, per-project activation; defaults retire into a profile card
3. Catalog entriesSingle entry per card with version list + stability channels (schema v2); auto-replace until then
4. Deprecation fixStaged A → B → C; Stage A shipped; metadata becomes distributable via refs/meta/*

Explicitly rejected (with reasons preserved)

  • Symlink-back materialization — breaks Windows/Cowork portability (analysis 82), content-hash drift detection, and team-shared cards.
  • Mutable published versions — destroys reproducibility; card link removes the ceremony instead.
  • Editing the store directly — it is a distribution cache, not a workspace (nix’s read-only store reasoning).
  • Manifest-carried deprecation — cannot deprecate a card that will never publish again (harness-skills’ exact situation).
  • Machine card for defaults — invests in machine-wide activation, the opposite of the decided principle.

Open fronts (from analysis 94, not yet scheduled)

  • Trust of instructions, not just code. Hooks require consent; skills — which an agent will follow — do not, and a card can bundle both a malicious instruction and the MCP server it abuses. Small blast radius today (own cards, one curated catalog); becomes the headline risk once community catalogs open. Path: apply-time content summaries now → catalog quality signals → signing before any default-registered community catalog.
  • Multi-machine registry sync. The registry is per-machine; sources, trust decisions, and catalogs do not sync (only cards with remotes do implicitly). Needs a documented remote-first pattern and eventually drwn store sync.
  • The evolutionary loop is the moat. Session signals + analyzer already exist; provenance makes skill variants attributable across projects. The meta ref is the reserved home for fitness / eval annotations — same mutable-metadata channel, no new machinery. No competitor is positioned to copy this.
