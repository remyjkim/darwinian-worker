Worker Mind Engineering Guide

The optional BeginningDB-backed state contract for one selected Darwinian Worker.

first supported contract | drwn 0.9.0 | reviewed 2026-07-13
Core model: a project selects at most one Worker. That Worker's locked Blueprint closure may contribute persona, beliefs, observations, or insights. If it contributes any of those, the Worker is Mind-capable; explicit provisioning creates its one optional Mind. A memory declaration is not the only way to enable a Mind.

1. Boundary And Ownership

A platform-assigned mindId is an identity and binding coordinate. It does not prove that a BeginningDB subtree has been provisioned. Runtime memory is not Card content and is not published, exported, or visibility-classified with persona and beliefs.

2. Capability Predicate

cardDeclaresMind(card) =
  card.persona.include is non-empty
  OR card.beliefs.include is non-empty
  OR card.memory.observations is present
  OR card.memory.insights is present

selectedWorkerDeclaresMind =
  any Card in the selected Worker closure satisfies cardDeclaresMind

Mind commands evaluate the selected closure and fail with MIND_CAPABILITY_NOT_DECLARED before contacting BeginningDB when the predicate is false. Installed but inactive Worker roots do not enable Mind operations.

Closed Memory Declaration

{
  "memory": {
    "observations": { "format": "jsonl" },
    "insights": { "format": "md" }
  },
  "harness": { "minVersion": "0.9.0" },
  "lastValidatedWith": "0.9.0"
}

A Card may declare observations only, insights only, both, or neither. Each present key has one fixed format. The object rejects unknown fields, generic formats, visibility, include/exclude lists, numbered kinds, and raw_data. An empty memory object contributes no capability. Persona-only and beliefs-only Workers remain valid Minds.

Version Floor

The command capability gate is selection-scoped, but the project lock floor is intentionally lock-wide. If any Card in any installed Worker closure is Mind-bearing, card.lock.store.minDrwnVersion must be 0.9.0; otherwise the base Worker floor is 0.8.0. This prevents selecting a locked alternative with an older CLI. Direct Worker deployment computes the same floor from the complete resolved closure.

3. Card Composition

CardRole
@darwinian/mind-toolsCanonical observations/insights declaration, five runtime Mind skills, semantic paths, placement rules, and deletion discipline; no persona or beliefs
@darwinian/mind-starterStandalone quickstart with the same memory declaration, generic persona, collaboration belief, and synced mind-tools content
Domain content CardPersona, beliefs, and domain skills; normally omits memory when composed with mind-tools
Worker BlueprintComposes mind-tools and domain Cards once into one deployable Worker

Persona entries compose in deterministic locked-closure order with Card and entry provenance fences. Beliefs remain separate attributed files. Their private | internal | public visibility controls Card publication and remote push. Memory is never seeded from Card-owned files; runtime skills and applications create it in BeginningDB.

4. Storage Grammar

/minds/<mindId>/
  persona.md
  beliefs/<card>/<entry>/BELIEF.md
  mind.json
  memory/
    observations/by-date/<yyyy-mm-dd>/<HHmm>-<ulid>.jsonl
    insights/by-date/<yyyy-mm-dd>/<HHmm>-<ulid>.md
    insights/by-topic/<topic>/<view>.md       # optional application view

/pool/
  observations/<yyyy-mm-dd>/<HHmm>-<ulid>.jsonl
  insights/<yyyy-mm-dd>/<HHmm>-<ulid>.md

Pool paths are canonical identity placements. Mind paths are views of the same BeginningDB inode, not byte copies. Every newly created entry uses an immutable, date-sharded pool path and a default by-date view. Applications may add semantic views under the correct kind, such as insights/by-topic, but may not introduce a new top-level memory kind.

One observations file represents one capture context and contains one JSON object per line. Because BeginningDB PATCH uses a previously observed offset, concurrent writers must not append to the same capture-context file. Each insight is one Markdown document with application-owned front matter carrying timestamp and observation provenance.

5. Strict Mind Index

{
  "schema": "drwn.mind-index",
  "schemaVersion": 1,
  "mindId": "mind_reviewer",
  "worker": {
    "card": "@team/reviewer-worker",
    "version": "1.0.0",
    "integrity": "sha256-..."
  },
  "cards": [
    {
      "card": "@team/reviewer-worker",
      "version": "1.0.0",
      "integrity": "sha256-..."
    }
  ],
  "persona": { "path": null, "entries": [] },
  "beliefs": { "entries": [] },
  "memory": {
    "observations": { "format": "jsonl" },
    "insights": { "format": "md" }
  },
  "ledger": [],
  "drwnVersion": "0.9.0"
}

The reader validates schema identity, version, provenance rows, persona/belief entries, ledger rows, and the closed memory object. Malformed supported state raises MIND_INDEX_INVALID; a missing/different identity or unsupported version raises MIND_INDEX_UNSUPPORTED. Both fail closed and direct the operator to a controlled reset and reprovision. There is no prototype reader, numbered alias, automatic translation, or dual write.

6. Operational Lifecycle

CommandContract
drwn worker mind provisionValidates the selected closure, then idempotently creates seed content and the strict index
drwn worker mind statusReports seed provenance and drift without mutation
drwn worker mind syncRebases Card seeds; DB edits win unless the operator explicitly uses force
drwn worker mind diffShows DB versus Card seed differences and unattributed persona content
drwn worker mind checkpointWrites attributable live seed edits to editable Card sources for review; unattributed content blocks checkpoint
drwn worker mind doctorDiagnoses binding, index, ledger, pool, placement, and unsupported-residue health
drwn worker mind pool retireHuman-confirmed delete_everywhere for one strictly parsed canonical pool entry

Memory Entry Lifecycle

  1. Remember: create a canonical pool entry with If-None-Match: *, then place its default Mind view.
  2. Share: place the same inode into another Mind view in the same owner filesystem.
  3. Forget: call DELETE ...?action=unplace for only the requesting Mind view. Missing views are idempotent no-ops; final-placement refusal is an error.
  4. Retire: a human explicitly invokes delete-everywhere through the guarded CLI command.

Bare DELETE is not an acceptable implementation of forget: BeginningDB can fall through to inode deletion when the path is the final placement. Agent-facing skills never receive delete-everywhere authority.

7. Reserved Raw Data

raw_data is a vocabulary reservation, not a feature. The first contract creates no raw-data directory and defines no format, ingestion, extraction, retention, sharing, or authorization behavior. Supporting it later requires a separately approved schema and complete lifecycle design.
