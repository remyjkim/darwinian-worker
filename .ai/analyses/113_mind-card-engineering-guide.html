Mind Cards

What they are, what they're made of, and how to set one up and use it — the engineering guide to giving a Darwinian worker a durable, DB-backed mind.

1What a mind card is

A mind card is an ordinary Darwinian capability card that carries the seeds of a worker's mind: persona entries (voice and operating style), belief entries (durable principles and decision rules), and the declaration of its memory layers. It is authored, versioned, published, and composed exactly like any other card — same manifest, same store, same lockfile — but its content has a second life: at provision time it is seeded into BeginningDB, where it becomes live, editable state that the worker's agents read and extend at runtime.

The one-sentence mental model: cards are the reviewed checkpoint lineage; BeginningDB is the living mind; the deploy API binds them per worker.
The unit of mind (get this straight first): a worker is not a mind, and a card is not a mind. A mind is optional state a worker has — workers are stateless by default and acquire a mind only when their composed stack includes at least one mind-bearing card and worker mind provision has run. Cardinality: a worker has at most one mind; a mind is seeded from one or more cards (stack-ordered); one card can seed many minds — the card is the genotype, each mind a phenotype that diverges through lived memory. The deploy API's mindId is a legacy field name for the worker's stable identity — minted for every worker, mind or not; the minds/<mindId>/ subtree exists only once provisioned. "Mind card" = any mind-bearing card; the substrate is @darwinian/mind-tools (skills + conventions, no seeds) and the solo quickstart is @darwinian/mind-starter (voice + collaboration + the synced skills).

Concretely, three things cooperate:

PieceOwnsMutability
The card (git + drwn store)Persona/beliefs seeds, memory layer declarations, agent skills, conventionsImmutable per version; changes are new semver versions, reviewed in git
The mind (BeginningDB, minds/<mindId>/)The live persona/beliefs the worker actually runs on, plus all memoryMutable, edited DB-first by humans and appended by agents
drwn CLI (worker mind …)The loop between them: seed, drift report, rebase, checkpoint—

2Why it exists

Two facts are in permanent tension. Cards are immutable, versioned artifacts — which is exactly right for a worker's definition: you want fleet reproducibility ("deploy 20 reviewers from @team/reviewer-mind@1.4.0"), diffable history, review-before-ship, and a known-good baseline to restore. But memory is mutable runtime state — a worker's observations and insights accrue every session and belong nowhere near an immutable artifact.

The mind card resolves the tension by splitting along it:

  • Persona and beliefs live in the card as reviewed seeds, and in the DB as live copies. The DB is the primary editing surface (product services edit files directly); the card is the checkpoint. Divergence is drift — reported, never auto-reverted. Edits worth keeping flow back into the card via checkpoint, get reviewed like any code change, and become the next baseline.
  • Memory never enters the card at all. The manifest only declares the layers; the content is DB-native from birth.
Why the checkpoint lineage matters: BeginningDB keeps no version history. Card versions are the only versioning the system has. A mind that never checkpoints has no recoverable past.

3Anatomy of the card

3.1 The manifest

{
  "name": "@team/reviewer-mind",
  "version": "1.2.0",
  "skills":  { "include": ["mind-read", "mind-remember", "mind-search"] },
  "persona": { "include": ["voice"],          "visibility": "internal" },
  "beliefs": { "include": ["code-quality"],   "visibility": "internal" },
  "memory": {
    "l4": { "format": "md"    },   // reflections / insights
    "l5": { "format": "jsonl" }    // observations   (l6 = raw data, reserved for V2)
  }
}
SectionRules
persona / beliefsinclude lists entry names (safe path parts). visibility ∈ private | internal | public and is required when include is non-empty. exclude/shared are rejected on cards.
memoryLayers restricted to l4 | l5 | l6; format ∈ md | jsonl | mixed. Declaration-only: include is rejected — memory entries are DB-native and never ship in a card.
skills / serversOrdinary capability content. Skills are how agents operate the mind (they wrap the bgdb CLI); MCP servers (e.g. Notion) ride along for ingestion tooling. Skills are card content — they are never uploaded to the DB.

3.2 The source layout

sources/@team/reviewer-mind/
  card.json
  persona/<entry>/PERSONA.md      ← drwn card source add-persona / remove-persona
  beliefs/<entry>/BELIEF.md       ← drwn card source add-belief  / remove-belief
  skills/<skill>/SKILL.md         ← standard skills authoring

Publish validation enforces that every declared entry has its PERSONA.md/BELIEF.md; card source doctor flags orphaned directories (bundled but undeclared) and missing files before you get that far. Locks that carry mind content raise the project's version floor: drwn ≥ 0.7.0 is required to read them.

4The DB side: the living mind

Every worker's mind is a subtree of its owner's BeginningDB filesystem. One filesystem per user hosts all of that user's minds plus one shared memory pool:

<owner filesystem>
├── minds/
│   └── <mindId>/                          mindId = the worker's stable id
│       ├── persona.md                     seeded: stack-ordered, provenance-fenced
│       ├── beliefs/<card>/<entry>/BELIEF.md
│       ├── memory/
│       │   ├── l4/…                       placement views of pool L4 entries
│       │   └── l5/by-date/…               placement views of pool L5 entries
│       └── mind.json                      seed ledger + index (drwn-managed)
└── pool/
    ├── l4/<yyyy-mm-dd>/<HHmm>-<ulid>.md    canonical reflection entries
    └── l5/<yyyy-mm-dd>/<HHmm>-<ulid>.jsonl canonical observation files

Memory = pool entries + placements

Entries are born in pool/ and appear in mind trees only as placements (BeginningDB's multiple-paths-to-one-file). Identity is the file, paths are views: the same observation can sit in by-date/ and a topic view, and can be shared into another mind's tree without copying — edit once, visible everywhere.

Layer semantics

L5 — observations: high-volume facts/events; one append-only .jsonl file per capture context (typically a session); line schema {ts, type, content, refs?, source?}.
L4 — reflections: distilled insights; one .md per reflection with front-matter ts / derivedFrom / topics.
L6 — raw data: reserved, don't create it yet.

Provenance fences (the checkpoint mechanism)

The seeded persona.md is a stack-ordered concatenation of every composed card's persona entries, each wrapped in fences:

<!-- drwn:persona:start card="@team/reviewer-mind" entry="voice" -->
… entry content — edit freely INSIDE the fence …
<!-- drwn:persona:end card="@team/reviewer-mind" entry="voice" -->

The fences are how checkpoint attributes a DB edit back to the owning card and entry. Content outside any fence cannot be attributed and blocks checkpoint — keep edits inside a fence, or move them into one.

mind.json — the seed ledger

Written at provision, updated by sync: which card versions were seeded, at which ETags, in which stack order. It drives the three drift states you'll see in status: in-sync, db-edited (live edits — preserved, informational), card-updated (a rebase is available).

5The lifecycle verbs

VerbDirectionWhat it does
card source add-persona / add-beliefauthor → cardScaffold entries with required --visibility; --dry-run --json previews.
card publishcard → storeValidates content dirs, snapshots the whole tree, semver-checks the bump. Throws if the version already exists — bump first.
card pushstore → remoteGated by the strictest visibility of the mind content (§10).
worker mind provisioncard → DBOne-time seed of minds/<id>/ from the active stack: fenced persona, belief copies, memory scaffolding, ledger. Idempotent — a provisioned mind is a no-op, never re-seeded.
worker mind status—Per-file drift table (ledger vs live ETags vs pinned versions).
worker mind synccard → DBRebase seeds onto the current card versions via compare-and-swap. Files edited in the DB are skipped and reported (DB wins); --force makes the card win; --dry-run plans. Memory is never touched.
worker mind diff—Per-entry DB-vs-card diff (fence-aware); shows outside-fence content.
worker mind checkpointDB → cardWrites live edits back into the local card source — you review the git diff and publish the next version. Fails closed on outside-fence content.
worker mind doctor—Binding reachability (unreachable = warning), ledger integrity, unplaced pool entries. Report-only.
worker mind pool retiredestroydelete_everywhere for a pool entry — human-only, confirmation-gated; agents get forget (unplace) instead.

The intended rhythm: seed once → live DB-first editing → occasional checkpoint → review → publish → sync. Drift is the steady state, not an error.

6Composition: blueprints and stacks

A worker is normally a blueprint — a card with "kind": "blueprint" — composing the mind substrate with content cards and carrying governance:

{ "name": "@remyjkim/chief-worker", "version": "0.1.0", "kind": "blueprint",
  "composedFrom": ["@darwinian/mind-tools@^0.1.0", "@remyjkim/chief@^0.1.0"],
  "permissions": { "requiresHumanApprovalFor": ["external_send", "file_write"] },
  "escalation":  { "humanOwner": "remy@curationlabs.ai" } }

Blueprints contribute zero skills and zero persona themselves — they are composition + governance specs. Members land in the lockfile beside the blueprint; two blueprints sharing @darwinian/mind-tools dedupe by name (first wins).

The substrate doctrine: the substrate is composed, never forked. Minds are always defined by their own content card. Worker blueprints compose [@darwinian/mind-tools, <content-card>, …]. @darwinian/mind-starter is the solo quickstart — apply it alone; composing it under a content card re-introduces the generic voice it exists to contain.
The no-expansion rule (the #1 operational surprise): neither worker stack use nor worker mind provision expands blueprints. activeWorkers is a plain name filter over the lock. If you activate only the blueprint, you activate nothing useful. Always list the members explicitly:
drwn worker stack use @remyjkim/chief-worker @darwinian/mind-tools @remyjkim/chief

Stack order matters twice: it is the persona composition order (which fenced section comes first in persona.md) and the skill materialization order. Switching workers locally = switching the stack and the target mind together (§7).

7Setup: from zero to a provisioned mind

7.1 Prerequisites

  • drwn ≥ 0.7.0 (earlier versions hard-reject persona/beliefs sections).
  • A reachable BeginningDB. For local dev, run the server binary: 
    BGDB_PROFILE=development BGDB_DATA_DIR=/tmp/bgdb BGDB_STORAGE=fs \
BGDB_BEARER_TOKEN=dev-token BGDB_BIND=127.0.0.1:8080  ./beginningdb

7.2 Publish the cards

drwn card publish reads sources from ~/.agents/drwn/sources/<scope>/<name> — copy your checkout there (or script it), then publish. Publishing an existing version throws; that's your cue to bump.

# one-time per card version
cp -r /path/to/mind-tools ~/.agents/drwn/sources/@darwinian/mind-tools
drwn card publish @darwinian/mind-tools
# … repeat for content cards and blueprints (members before blueprints is tidy)

7.3 Lock them into a project

# card apply accepts MANY specs; card add accepts exactly ONE
drwn card apply @remyjkim/chief-worker@^0.1.0 @darwinian/mind-tools@^0.1.0 @remyjkim/chief@^0.1.0

7.4 Connect and provision

The mind connection is environment-driven. Required: BGDB_BASE_URL and BGDB_TOKEN (+ BGDB_TENANT_ID in direct mode). The mind id comes from --mind-id or, as shorthand, BGDB_PATH_PREFIX=minds/<mindId> (optional — and not a security boundary, see §10).

# .env.chief — one file per worker keeps stack + mind switching atomic
export BGDB_BASE_URL=http://127.0.0.1:8080
export BGDB_TOKEN=dev-token
export BGDB_TENANT_ID=1
export BGDB_PATH_PREFIX=minds/mind_chief_local

# provision (seed) — idempotent
drwn worker stack use @remyjkim/chief-worker @darwinian/mind-tools @remyjkim/chief
source .env.chief
drwn worker mind provision
drwn write            # materialize skills/MCP for your coding agents
drwn worker mind status --json   # everything in-sync
The seeding footgun: provision reads content from the currently active stack. Run it with the wrong stack active and you seed the wrong persona into the mind. The per-worker env file + stack command pair above is the discipline that prevents it — always switch both together.

7.5 Multiple workers, one machine

Repeat 7.4 per worker with its own env file and mind id. Each worker gets its own minds/<id>/ subtree; the shared pool/ lets them share memory entries deliberately (placements), never accidentally.

8Using the mind at runtime

Agents operate their mind through the card's skills, which wrap the bgdb CLI (env-configured by the same BGDB_* variables):

SkillDoesUnder the hood
mind-readLoad persona + beliefs at session startbgdb fs cat (fences are provenance markers — ignore, don't delete)
mind-rememberRecord an observation (L5) or reflection (L4)Create the pool entry, then place it into the mind's memory view
mind-shareShare an entry with another mindPlace the same entry into the other mind's view — no copies, ever
mind-forgetRemove an entry from this mind's viewUnplace only. Refuses if it would be the last placement (that's retirement — human-only)
mind-search"What do I remember about…"bgdb search, one path-prefix per request

Server-side integrations (a backend accruing memory on a worker's behalf, as in believer-interview's chief) speak the same small BeginningDB HTTP surface directly: PUT /v1/fs with If-None-Match: * for atomic creates, ETag compare-and-swap for edits, and append as stat → PATCH with Content-Range: bytes <size>-. Append is not atomic — the convention that keeps it safe is one writer per entry file (a session appends only to the file it created). The client is ~150 lines of pure fetch, Workers-safe.

9Deployed workers

drwn worker deploy <blueprint-ref> --name chief gives each deployed worker a stable mind id from the deploy API (it outlives redeploys and rollbacks — memory persists across versions). The CLI caches the non-secret binding coordinates in ~/.agents/drwn/mind-bindings.json; tokens are never persisted. The container receives the same BGDB_* environment, so the skills work unchanged in deployed workers. (Full server-side rollout is tracked in darwinian analysis 107; until your deploy API implements it, the CLI degrades gracefully and tells you to set BGDB_* yourself.)

10Visibility, isolation, deletion

Visibility & the push gate

Persona/beliefs sections carry visibility; the card's effective visibility is the strictest across sections. drwn card push refuses to push visibility-bearing content to a remote whose visibility is unknown or less restrictive — for a normal https remote you must state it: drwn card push @x/mind --remote-visibility public. Overriding (--unsafe-push-public) is loud and audited.

Isolation (V1, be honest with yourself)

Tokens are filesystem-scoped: a worker's token technically reaches its sibling minds and the whole pool. BGDB_PATH_PREFIX is ergonomic addressing, not a security boundary. Per-mind separation is skill/CLI convention until BeginningDB's ReBAC-on-VFS enforcement lands. Acceptable because all colocated workers belong to one user; org-shared pools are explicitly gated on real enforcement.
Deletion policy (no history = no undo): agents may only unplace from their own views. Destroying a pool entry (worker mind pool retire) removes it from every mind permanently and is a confirmation-gated human verb. Never wire delete_everywhere into an agent.

7checkpoint fails: content outside fencesMove the prose inside the owning entry's fence (or delete it), rerun. Unattributable content can't be checkpointed
8Concurrent appends clobbered an L5 fileAppend is stat→PATCH, not atomic — one writer per entry file; each session creates its own
9Old drwn can't read the projectLocks with mind content set a 0.7.0 floor; upgrade drwn
