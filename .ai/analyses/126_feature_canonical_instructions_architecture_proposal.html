Canonical Instructions Projection for drwn

One set of composed worker instructions, three delivery layers, four coding agents: a managed AGENTS.md block for the main conversation, per-target sub-worker surfaces, and an instructions-skill floor everywhere else — all rendered from the single composer drwn already ships.

01

Drivers

A shipped pipeline with a missing last hop, and a new hard requirement on how the hop must be built.

Architecture driver

Validated worker instructions never reach a local harness — and any fix must now serve all four supported agents

Cards declare an instructions field that drwn validates, locks, and composes into a canonical artifact on every write, but no local harness surface consumes it — authors get a silent no-op (issue I24, analysis 100). Three constraints govern the fix: cross-CLI parity across Claude Code, Codex, Cursor, and OpenCode is a hard requirement (Remy, 22 Jul); projected instructions from a third-party card are instruction injection, so an ungated design is not acceptable (I24 criterion 4); and the existing composer must remain the only instruction source (I24 criterion 5).

02

Current state

Everything up to delivery already works; delivery itself is a workaround.

Verified in repo · 22 Jul

The canonical artifact is composed on every write and consumed only by deploy

Shape validation lives in card-manifest.ts; sync-worker.ts composes explicit card instructions, Blueprint identity, and skill bodies into instructions.md for every applied root (lines 283, 356). Deployed workers consume it; cli/ contains zero references to AGENTS.md, CLAUDE.md, or any instruction surface. The docs contract (project-worker-v1.md:108) promises the projection that was never built.

Workaround in production

Workers ship their instructions as a pull-strength skill, and authors cannot even set the field

The Deep Research panel workers deliver their instructions as a bundled instructions-skill — pull-based, probabilistic triggering at user-message strength. Meanwhile drwn card source set has no --instructions flag, so the field the manifest advertises is only reachable by hand-editing card.json. Both facts are documented and agreed in the I24 team discussion.

03

Target architecture

Three delivery layers, one composer, per-target adapters in the pattern the descriptor table already uses for MCP, skills, and hooks.

1Main worker instructions

AGENTS.md managed block, consent-gated

The composed worker instructions render into a marker-delimited, hash-sentineled drwn block inside repository-root AGENTS.md — user sections untouched, drift detected, removal clean, using the managed-block idiom already contract-tested for opencode.json and .gitignore. Codex, OpenCode, and Cursor consume root AGENTS.md natively; Claude Code needs only a one-line .claude/CLAUDE.md import adapter (analysis 123). Projection requires instruction consent through the existing card-trust surface, auto-satisfied for first-party sources (doc 125 B-Q1).

2Sub-worker instructions

A+O×N — one authored sub-worker file, four target surfaces

Cards ship agents/<name>.md once; a descriptor-table field projects it per target: markdown passthrough to .claude/agents/, .cursor/agents/, and .opencode/agents/ (where the body is the sub-worker's actual system prompt), and a markdown-to-TOML adapter for .codex/agents/<name>.toml mapping the body to developer_instructions — the same per-target format-writer class the MCP surface uses for json-merge versus toml-merge. Deploy consumes the identical file via instructions: {path}, so local and deployed instructions cannot diverge.

3Universal floor

Instructions-skill, generated from the same composition

A <worker>-instructions skill generated from the composed artifact (never hand-mirrored) keeps delivery alive wherever a push surface is absent, disabled, or broken. The AGENTS.md block references it by name, converting probabilistic pull toward deterministic load. Scaffolding (card new --instructions-skill) and a publish lint — worker cards with neither instructions nor an instructions-skill warn — close the original silent no-op.

04

Migration

Two phases, each independently shippable; the drwn-as-launcher convergence (I24 option F) stays an explicit horizon, not a phase.

1Phase 1 · main instructions

AGENTS.md block, adapter, consent, authoring

Extract the shared managed-block helpers; add the instructions projection surface and a sync-instructions step reusing the existing composer; write the AGENTS.md block and the guarded .claude/CLAUDE.md adapter; gate on instruction consent via card trust; add --instructions to card source set and fix the contract line; doctor checks for block staleness and adapter presence. Roughly ten TDD tasks in the established test patterns, no schema migrations (doc 124 §4 as amended by doc 125 Part C).

2Phase 2 · sub-worker instructions

Sub-worker content type and per-target writers

Add the sub-worker card content type, a sub-worker-surface descriptor field, markdown writers for claude/cursor/opencode and the TOML adapter for codex, plus the instructions-skill scaffold and lint. Entry is gated on closing the harness verify items: cursor agents format, codex spawn honoring custom agent files (openai/codex #26868, #14579), and the live-verification pass the opencode target work established as precedent.

Risk

The two risks that shape the design, with the evidence behind each.

Material risks, evidence, and how the architecture answers them
SignalEvidenceInterpretation
Projected instructions are an injection channel with repo-wide blast radiusI24 criterion 4: "a projected system prompt from a third-party card is silent prompt injection … ungated is not an answer." The AGENTS.md block lands in a committed file every collaborator's agent reads. The skill channel is already consent-free today — a third-party card's skill injects instructions whenever it triggers.Consent-gate projection through the existing card-trust machinery (hook-consent precedent), auto-satisfied for first-party sources, CI-enforceable via strict mode. Document the consent-free skill sibling honestly rather than treating the question as new (doc 125 B-Q1, B-Q7).
Harness surfaces are young and move underneath the designCodex multi_agent is stable and enabled on the installed 0.144.6, but upstream issues report custom agent TOMLs not honored on spawn in some versions (#26868, #14579). Cursor's agents-file format is undocumented. OpenCode 1.17.13 dropped project-level .claude/skills discovery — a doc-121 assumption falsified only by live verification.Per-target adapters isolate breakage to one writer; verify items gate phase 2 rather than block phase 1; the instructions-skill floor keeps delivery alive when a push surface regresses. Re-verify against real binaries before each phase, as the opencode target work established.
