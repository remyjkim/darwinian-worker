# v0.7.0 Release Readiness Design

## Goal

Bring the repository's implemented `0.7.0` state to a truthful, reproducible release while preserving the standalone security boundary of `drwn-command-bridge` and the independent lifecycle of `darwinian-worker-skills`.

## Approved Approach

The CLI package remains Bun-native for `0.7.0`. Its npm entrypoint is TypeScript with a Bun shebang and the command implementation uses Bun APIs, so user-facing installation docs and package metadata must state that Bun is a runtime prerequisite. A Node-only distribution remains a later packaging project because the current Node bundle fails once commands reach Bun APIs.

The skills checkout is normalized to the tracked `darwinian-worker-skills/` gitlink. The clean legacy-named checkout points at the exact pinned commit, so it can be reused without changing the parent repository's recorded submodule revision.

`drwn-command-bridge` remains a standalone Node-targeted package within this repository. Root CI installs its isolated dependency graph and verifies its typecheck, tests, build, and npm tarball. Its package metadata and release documentation must be complete before it is committed. Publication is gated on a native macOS end-to-end smoke through Claude Desktop or an equivalent MCP stdio client; Linux and Windows limitations remain explicit.

The existing mind-substrate and bridge design artifacts are committed separately from runtime/package changes. Stale forward-looking paths or status statements are corrected, while dated historical records remain historical.

## Alternatives Considered

1. Convert `darwinian@0.7.0` to a Node bundle now. Rejected for this release because the generated bundle still contains direct Bun API calls and fails on real commands under Node.
2. Keep the bridge outside repository CI. Rejected because root `bun test` discovers nested tests and a fresh clone otherwise lacks the bridge's isolated dependencies; explicit package checks make the boundary reproducible.
3. Publish the bridge before a native smoke. Rejected because its value is security-sensitive host execution, and simulated wrapper tests alone do not establish native integration behavior.

## Verification And Release Flow

Each logical change is committed independently. Before integration, run both package verification suites, inspect both npm tarballs, and confirm the CLI package excludes the bridge and submodule. Merge the release branch through the repository's normal review path. Only a clean `main` commit with all checks green may receive the annotated `v0.7.0` tag; the tag-triggered workflow owns the `darwinian` npm publish and GitHub release.

Machine-wide Darwinian Minds defaults are explicitly out of scope because another coworker owns that repair.
